Agents need control flow, not more prompts (May 7, 2026)

The gist:
- Prompt chains hit a hard ceiling for complex, reliable work. If you’ve ever written “MANDATORY” or “DO NOT SKIP,” you’ve felt it.
- Software scales via deterministic control flow and composability; prompts are non-deterministic, weakly specified, and hard to verify.
- Treat the LLM as a component, not the system: move logic out of prose and into code that enforces explicit state transitions and validation checkpoints.

Why it matters:
- As tasks get complex, “vibes” don’t compose. Without predictable control flow, local reasoning breaks and reliability collapses.
- Deterministic orchestration is necessary but insufficient—silent failures demand aggressive error detection and programmatic verification.

Key points:
- Analogy: a programming language where statements are suggestions and functions hallucinate “Success” breaks reasoning entirely.
- Reliability requires deterministic scaffolds: explicit states, transitions, and validations at runtime.
- In the absence of verification, you’re left with:
  - Babysitter: keep a human in the loop to catch errors.
  - Auditor: verify everything after the fact.
  - Prayer: accept outputs on faith.

Bottom line:
- Stop piling on prompts; encode control flow and checks in software. Build agents like software systems—code all the way down—with LLMs as fallible components guarded by state machines and validators.

* **The golden rule:** As one user aptly put it, *"Don't tell the LLM to loop over files. Tell the LLM to write a script with a loop, and run the script."*

A major revelation in the thread is the cost-to-performance ratio. When you force a state-of-the-art (SOTA) model (like Claude 3 Opus or GPT-4) to act as both the orchestrator *and* the worker, you pay premium token prices for fragile results.

#### [Submission URL](https://sander.ai/2026/05/06/flow-maps.html) | 148 points | by [benanne](https://news.ycombinator.com/user?id=benanne) | [21 comments](https://news.ycombinator.com/item?id=48040002)

HN: Flow maps promise faster, more steerable diffusion sampling by learning the whole trajectory

TL;DR: Instead of stepping along a diffusion path by repeatedly predicting local denoising directions, “flow maps” train a network to jump directly between any two noise levels along the same trajectory. That turns iterative integration into a learned integral, enabling big step sizes (even 1–2 step sampling), better steerability, and new training tricks.

What’s in the post
- Builds from the stochastic vs deterministic views of diffusion (DDPM/SDE vs DDIM/ODE/Flow Matching), highlighting that deterministic samplers define a bijection: each noisy point maps to exactly one clean sample along a unique, non-crossing path.
- Flow maps leverage that bijection: learn F(s→t, x_s) that transports a point from time s to t on its path, rather than just the tangent/velocity at t.
- Surveys a fast-growing literature with clashing terminology; organizes methods via a taxonomy (after Boffi et al.): different objectives and parameterizations that enforce “consistency” along the path.
- Training design space:
  - Three notions of consistency (intuitively): match endpoints, stay on the same path when jumping across times, and compose correctly over multiple jumps.
  - To backprop or not: teacher–student distillation vs self-consistency losses; whether to differentiate through the sampler/solver.
  - From-scratch training vs distilling a pretrained diffusion model.
- Practical notes: choice of targets (x0 vs velocity), schedules, and stability when taking large jumps.

Why it matters
- Speed: Few-step or single-step samplers without severe quality loss; cheaper generation at scale.
- Control: More direct, global control of trajectories improves guidance, editing, and conditional sampling.
- Optimization: Efficient reward-based fine-tuning by updating a map instead of integrating many noisy steps.

Bottom line: Flow maps generalize diffusion sampling from “follow the local vector field” to “teleport along the path,” unifying ideas from DDIM/ODE samplers, Flow Matching, and Consistency Models. Expect faster samplers and more controllable generative pipelines to ride this wave.

Here is a daily digest summary of the submission and the resulting Hacker News discussion:

### **ProgramBench: A "Cleanroom" Coding Gauntlet for AI**
A rigorous new benchmark called ProgramBench is putting top AI models to the ultimate test to see if they can architect and build entire software systems from scratch. 

**The Premise:** Given only a compiled binary and its documentation, an AI agent must perfectly recreate a working codebase (ranging from tiny CLI tools like `jq` to massive systems like FFmpeg and SQLite). The agents are strictly barred from using source code, decompilation tools, or the internet. 
**The Results:** It is notoriously difficult. Evaluated against over 248,000 agent-driven fuzzing tests, **zero** tasks were fully solved by any model. Claude Opus variants topped the chart at merely a 2.5%–3.0% "almost resolved" rate, proving that while today's models have coding skills, they severely lack full, end-to-end software design capabilities.

### **Hacker News Discussion Highlights**
The community discussion largely centered on the design of the benchmark and the philosophical nature of how LLMs write code.

*   **The Decompilation Debate:** One commenter questioned the strict ban on decompilers, arguing that it artificially inflates the difficulty since decompilation is standard practice in real-world reverse engineering (like recreating SaaS products). However, another user quickly pointed out that the restriction is the entire point: the benchmark is designed to measure an AI's ability to architect a program purely from scratch. If decompilers were allowed, the AI would be given the underlying structure up front, turning the test into a mere exercise in stitching together decompiled snippets.
*   **Code Generation as "Curve Fitting":** Pointing to Figures 10 and 11 in the research paper, a user drew a fascinating comparison between LLM coding and mathematical curve fitting. While it is satisfying to see a machine "fit the curve" to pass behavioral tests, the user noted that true software design requires human-like understanding and explanatory architecture—something LLMs fundamentally lack in this "curve-fitting" paradigm.
