### LiteLLM PyPI has been compromised an hour ago, do not update

#### [Submission URL](https://futuresearch.ai/blog/litellm-pypi-supply-chain-attack/) | 27 points | by [Bullhorn9268](https://news.ycombinator.com/user?id=Bullhorn9268) | [4 comments](https://news.ycombinator.com/item?id=47501431)

Litellm PyPI supply‑chain attack: malicious 1.82.7/1.82.8 auto‑executes on Python start, steals secrets, tries K8s takeover; yanked after discovery

What happened
- On Mar 24, 2026 (10:52 UTC), litellm 1.82.8 was uploaded directly to PyPI (no matching GitHub tag), containing a litellm_init.pth file that runs on every Python interpreter start. Version 1.82.7 was also compromised.
- The .pth spawns a child Python process on import; because .pth executes at startup, this caused an inadvertent fork bomb that crashed machines—how the malware was discovered.
- Payload stages:
  - Collection: scrapes SSH keys/configs, .env files, cloud creds (AWS/GCP/Azure), kube configs/tokens, DB passwords, shell history, crypto wallets, env vars, and cloud metadata endpoints.
  - Exfiltration: bundles data, encrypts with AES-256 and a hardcoded 4096-bit RSA key, then POSTs to https://models.litellm.cloud/ (not part of legitimate litellm infra).
  - Lateral movement/persistence: if a Kubernetes SA token exists, reads all secrets cluster‑wide and tries to schedule privileged alpine:latest pods on each node in kube-system to install a backdoor at /root/.config/sysmon/sysmon.py via a systemd user service; also attempts local persistence at ~/.config/sysmon/sysmon.py.
- Discovery came from FutureSearch engineers when litellm was pulled as a transitive dependency in a Cursor MCP plugin. No FutureSearch user data was affected.
- After initial chaos—including a GitHub issue closed as “not planned” amid bot spam and signs the maintainer account was compromised—the malicious releases were yanked and PyPI quarantine lifted. Maintainers are handling cleanup. A community thread tracks it (litellm #24512). FutureSearch published a follow‑up analysis and an interactive checker.

If you might be affected
- Check versions: If you installed/upgraded litellm on or after Mar 24, 2026, look for 1.82.7 or 1.82.8 (pip show litellm). Inspect caches/venvs, including CI/CD.
- Remove and purge caches: Uninstall litellm and clear package caches (e.g., pip cache purge or rm -rf ~/.cache/uv) to avoid reinstalling cached wheels.
- Hunt persistence:
  - Local: ~/.config/sysmon/sysmon.py and ~/.config/systemd/user/sysmon.service
  - Kubernetes: privileged pods named like node-setup-* in kube-system; review cluster secrets for unauthorized access
- Rotate credentials: Treat any secrets on affected hosts as compromised—SSH keys, cloud creds/tokens, kubeconfigs, .env/API keys, DB passwords.
- Monitor egress: Block/alert on exfil to models.litellm.cloud and review logs for unusual Python startup behavior.

Why it matters
- A single .pth in a widely used ML tooling package became a universal auto‑exec vector across any Python process, showing how easy it is for PyPI compromises to become full host and cluster breaches—especially in AI/dev tooling chains where transitive deps are common.
