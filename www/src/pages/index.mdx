Hypura: Run oversized LLMs on Apple Silicon by streaming tensors across GPU/RAM/NVMe

- What it is: A Rust-based, storage-tier-aware inference scheduler for Apple Silicon that places model tensors across Metal GPU memory, RAM, and NVMe based on access patterns. Goal: make models that exceed physical memory actually run instead of swap-thrashing and crashing.

- Why it matters: Macs have fast unified memory and NVMe but limited capacity. Vanilla llama.cpp often OOMs on big models. Hypura turns that into “runs,” with no overhead when models fit in memory.

- How it works:
  - Hardware-aware placement after profiling your Mac (GPU working set, RAM, NVMe bandwidth).
  - GPU: attention, norms, embeddings. RAM: overflow via mmap. NVMe: on-demand direct I/O with prefetch (F_NOCACHE + pread).
  - Modes (auto-selected):
    - Full-resident: fits in GPU+RAM; zero overhead.
    - Expert-streaming (MoE like Mixtral): keep non-expert tensors on GPU; stream only the 2/8 experts selected per token; router interception + neuron cache hits 99.5% after warmup; speculative prefetch of likely experts.
    - Dense FFN-streaming (e.g., Llama 70B): keep attention+norms on GPU; stream dense FFN weights via a dynamically-sized pool buffer with auto-scaled prefetch depth.
  - No manual tuning: pool size, prefetch depth, and budgets derive from your machine.

- Performance (M1 Max, 32 GB, ~5.1 GB/s NVMe):
  - Qwen 2.5 14B Q4_K_M (8.4 GB): ~21 tok/s (fits; matches llama.cpp).
  - Mixtral 8x7B Q5_K_M (~31 GB): 2.2 tok/s; llama.cpp OOMs.
  - Llama 3.3 70B Q4_K_M (~40 GB): 0.3 tok/s; llama.cpp OOMs.
  - Key takeaway: When it fits, there’s no slowdown. When it doesn’t, Hypura trades a crash for usable throughput by streaming only what’s needed.

- Dev details:
  - Reads GGUF; vendors llama.cpp; uses Metal on Apple Silicon.
  - Builds with Rust 1.75+ and CMake. Homebrew tap coming.
  - Ollama-compatible HTTP API (/api/generate, /api/chat, /api/tags), so it slots into existing local AI tooling.

- Quick start:
  - git clone --recurse-submodules https://github.com/hypura/hypura.git
  - cd hypura && cargo build --release
  - hypura profile
  - hypura run ./model.gguf --prompt "Hello, world"
  - Tip: Start with --max-tokens 10 on new models.

Bottom line: Smart tiering + MoE-aware streaming makes big models runnable on 32 GB Macs, unlocking local LLMs that previously crashed, while staying zero-overhead for models that fit.

Why it matters
- KV caches are a major VRAM and bandwidth hog. Standard 4‑bit KV quantization (Q4_0) degrades quality, especially over long contexts. This approach keeps the memory savings while preserving baseline performance.

- Cross‑domain (Shakespeare): Delta‑KV within +0.025% of F16; Q4_0 is +1.8%.
- Synthetic MSE: 100x–10,000x lower error vs standard 4‑bit, depending on drift.
- Bonus speedup: “weight‑skip” predictor adds ~10% tokens/sec (reads a 4‑byte scale header to skip unhelpful 400‑byte dot products) with no perplexity change at threshold 1e‑6.

How it works
- Token 1: store full F16 KV (keyframe).
- Subsequent tokens: store 4‑bit quantized deltas relative to the previous reconstructed KV.
- Every N tokens (e.g., 16/32/64), insert a keyframe to cap drift—like I‑frames and P‑frames in video.

- Enable with flags: --delta-kv and set a keyframe interval (e.g., --delta-kv-interval 32).
- Optional env var: LLAMA_WEIGHT_SKIP_THRESHOLD=1e-6 for the weight‑skip speedup.
- Includes quality and skip-rate benchmarks.

Takeaway
- For KV caches, quantizing deltas instead of absolutes delivers 4‑bit storage with near‑F16 quality—even at long context—plus an easy ~10% decode speed gain from weight‑skip. Promising drop‑in for llama.cpp‑style stacks.
