Vera: a verification-first programming language for LLMs (compiles to WebAssembly)

- What it is: A new language built for code written by large language models. Vera replaces names with structural references (e.g., @Int.0), makes contracts mandatory, and types effects. Programs compile to WASM and run on the CLI or in the browser.

- Why it exists: LLMs struggle less with syntax and more with global coherence—maintaining invariants, tracking state, and especially handling names. Vera’s premise: the model doesn’t need to be right, it needs to be checkable.

- How it works:
  - No variable names; all bindings are positional (avoids naming errors).
  - Mandatory contracts: requires/ensures proved statically by an SMT solver; violations are type errors (e.g., divide-by-zero).

- Getting started: Python 3.11+, pip install -e .; try vera check examples/absolute_value. For LLM demos: set VERA_ANTHROPIC_API_KEY and run examples/inference.vera.

- Open questions: human readability without names, authoring overhead of contracts, and how the approach scales to larger systems.

Structured Output Benchmark (SOB): finally, a test for whether your JSON is right—not just valid

What’s new
- SOB benchmarks how well LLMs extract and populate structured data across text, images (OCR), and audio—without conflating extraction quality with parsing, schema validation, vision, or ASR.
- It moves beyond “does it parse?” to “are the values correct, complete, and grounded?”

Why it matters
- In real pipelines, a valid JSON blob with wrong or missing values silently breaks downstream systems. SOB centers Value Accuracy—the share of fields you can trust in production—while still tracking parse and schema health.

How it works
- Three modalities, one scoring harness:
  - Text: HotpotQA passages (5,000)
  - Image: olmOCR-bench docs (209)
  - Audio: AMI meetings (115)
- To isolate structured-output skill, image/audio are normalized to text before scoring—so results reflect extraction and schema handling, not OCR/ASR quality.
- Seven metrics per record:
  - Value Accuracy (primary), JSON Pass Rate, Type Safety, Structure Coverage, Path Recall, Faithfulness (groundedness), Perfect Response (all leaves exactly right)
- Scoring gates:
  - Hardening: if JSON doesn’t parse, all semantic scores zero for that record
  - Coverage: missing paths count as wrong; no credit for values you didn’t return
- Difficulty weighting: schemas labeled easy/medium/hard; leaderboard is complexity-weighted (1x/2x/3x).

- Structural metrics (parse, types, coverage, paths) cluster near the ceiling for most models; the real separation is in Value Accuracy and Perfect Response.

- Perfect Response rates remain far below structural metrics, underscoring how hard end-to-end correctness is on nested and hard schemas.

What it fixes vs prior benchmarks
- Doesn’t collapse everything to “valid JSON” or single-domain accuracy
- Separates parse/structure/value errors so you know what actually failed
- Weights hard, nested schemas more heavily to reflect real difficulty

For builders
- Track Value Accuracy and Path Recall alongside JSON/schema checks; that’s your “trust without human review” signal.
- Expect near-perfect parsing but variable fact correctness—budget review for hard schemas and long-tail fields.
- If your workload is multi-modal in the wild, SOB’s normalization shows how well models will populate fields once OCR/ASR are solved.

SOB is a practical, production-minded benchmark: it tells you not just whether a model can emit valid JSON, but whether it fills the right keys with the right, source-grounded values—especially on complex, nested schemas.

Written in Micro Planner and Lisp on a PDP-6 (later with a full 3D rendering at the University of Utah), it showcased how far symbolic AI could go inside a tightly scoped domain.

What it did
- Parsed simple English (~50-word vocabulary) to move and arrange objects in a virtual world with basic physics.
- Tracked context and resolved references (“the cone” refers to the one just discussed), and asked clarifying questions when ambiguous.
- Answered questions about state and history (“What does the box contain?”, “When did you pick it up?”).
- Reasoned about possibilities and limitations (e.g., blocks can stack; triangles can’t) based on attempts in the world.
- Learned user-defined concepts and names (“a steeple is a small triangle on a tall rectangle”) and could build new ones.
- Explained its actions and goals step by step when asked “Why?” and “How?”

Why it still matters
- A landmark demo of grounded language understanding: language tied to perception and action.
- Transparent reasoning and goal chains—something today’s LLMs often struggle to expose.
- A reminder of the power of constrained domains: strong competence emerges when the world is small, rules are explicit, and symbols are grounded.

#### [Submission URL](https://www.augmentcode.com/blog/how-to-write-good-agents-dot-md-files) | 133 points | by [gmays](https://news.ycombinator.com/user?id=gmays) | [43 comments](https://news.ycombinator.com/item?id=47938417)

HN Digest: What actually works in AGENTS.md

TL;DR: Internal study finds most AGENTS.md docs either don’t help or actively hurt coding agents. The winners are short, procedural, and decision-focused; the losers are long, warning-heavy, and encyclopedic. Done right, the gains feel like upgrading the model tier; done wrong, it’s worse than having no doc.

How they tested
- Pulled dozens of AGENTS.md files from a large monorepo.
- Replayed real internal PR tasks using an eval suite (AuggieBench), running each task with and without AGENTS.md.
- Filtered for single-module scopes where agent docs could plausibly matter; compared against the “golden” PRs reviewed by senior engineers.

Key findings
- Same file, opposite effects: The very same AGENTS.md boosted best_practices by 25% on a routine bug fix but cut completeness by 30% on a complex feature task—because it triggered overexploration.
- Size matters: Best results came from a 100–150 line main AGENTS.md plus a few focused reference docs. Longer than that and gains reversed, especially in modules ~100 core files.
- Progressive disclosure wins: High-level guidance in the main file with tightly scoped references the agent can load on demand outperformed “cover everything here” docs.
- Write workflows, not essays: Numbered, step-by-step procedures moved agents from failing to finishing. Example: a 6-step deployment cut missing wiring files from 40% to 10%; correctness +25%, completeness +20%.
- Decide up front: Simple decision tables (e.g., React Query vs Zustand) improved adherence to conventions; best_practices +25%.
- Show, don’t tell: 3–10 line real code snippets/templates boosted code_reuse by ~20%. Too many examples led to mispatterning.
- Domain rules help—sparingly: Specific, enforceable rules (e.g., use Decimal for money) help; stacking dozens invites overexploration.
- Pair every “don’t” with a “do”: Prohibitions alone made agents cautious and meandering; pairing with concrete alternatives kept them moving.
- Modularity applies to docs too: Isolated submodules with modular docs performed best.

What failed
- Overlong, comprehensive AGENTS.md files.
- Walls of “don’ts” with no “dos.”
- Reference sections that pull the agent into reading sprees, spawning unnecessary abstractions and incomplete work.

Practical recipe
- Keep the main AGENTS.md ~100–150 lines.
- Include: 1–3 core workflows, 1–2 decision tables for common forks, a few real code snippets/templates, and a handful of specific, enforceable domain rules.
- Defer details to small, well-scoped reference files.
- For every “avoid X,” add “use Y.”
