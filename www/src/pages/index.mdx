ChatGPT ads, reverse‑engineered: SSE-injected units, contextual targeting, and a merchant SDK tied together with Fernet tokens

- How ads appear: While ChatGPT streams model output over SSE from chatgpt.com/backend-api/f/conversation, the backend injects typed single_advertiser_ad_unit objects alongside the assistant’s tokens. Each unit includes brand metadata, creative hosted on bzrcdn.openai.com, and a carousel card with a target URL. target.open_externally=false opens links in ChatGPT’s in-app webview so OpenAI can observe post‑click navigation.

- Contextual targeting observed: The same account saw different brands matched to different chat topics (e.g., Beijing travel → GetYourGuide/Axel, NBA playoffs → Gametime, slides/productivity → Canva, fashion → Aritzia, generic food query → Grubhub). The researcher didn’t find evidence confirming or denying use of prior conversation history.

- Four-token attribution chain (all Fernet; AES‑128‑CBC + HMAC‑SHA256):
  - ads_spam_integrity_payload: shipped in the SSE payload only; appears to validate server-side against forged clicks.
  - oppref: added to the click URL and persisted by the merchant SDK in a first‑party cookie __oppref (TTL ~30 days); used as the forward attribution token on subsequent events.
  - olref: on the click URL but not stored by the SDK observed; likely for impression/outbound logging on OpenAI’s side.
  - ad_data_token: base64-wrapped JSON containing another Fernet token; reconciled server-side at click.

- Token timing: Fernet’s header leaks a timestamp (version byte + 8‑byte big‑endian Unix time), letting you infer mint time without the key. One captured click showed a 95‑second lag between token mint and the merchant page load.

- Merchant-side loop: Landing pages load an OpenAI SDK (oaiq.min.js v0.1.3) from bzrcdn.openai.com, initialize with a merchant pixel ID, read ?oppref= from the URL, write __oppref and a probe cookie __oaiq_domain_probe, then POST events (e.g., contents_viewed) to https://bzr.openai.com/v1/sdk/events with st=oaiq-web&sv=0.1.3.

- IDs and assets: Advertiser accounts look like adacct_<32-hex>. Brand favicon and ad images are served from bzrcdn.openai.com, indicating OpenAI hosts creatives.

- Practical takeaways for users:
  - Blocking: bzrcdn.openai.com and bzr.openai.com will suppress SDK loads and event posts.
  - Inspect cookies after clicking in ChatGPT’s webview: __oppref and __oaiq_domain_probe.
  - Expect links to open in-app unless target.open_externally is true, enabling additional post-click tracking.

All findings come from observed, consented mobile traffic on a research fleet; internal logic beyond the wire data is inferred but not confirmed.

### A good AGENTS.md is a model upgrade. A bad one is worse than no docs at all

#### [Submission URL](https://www.augmentcode.com/blog/how-to-write-good-agents-dot-md-files) | 133 points | by [gmays](https://news.ycombinator.com/user?id=gmays) | [43 comments](https://news.ycombinator.com/item?id=47938417)

HN Digest: What actually works in AGENTS.md

TL;DR: Internal study finds most AGENTS.md docs either don’t help or actively hurt coding agents. The winners are short, procedural, and decision-focused; the losers are long, warning-heavy, and encyclopedic. Done right, the gains feel like upgrading the model tier; done wrong, it’s worse than having no doc.

How they tested
- Pulled dozens of AGENTS.md files from a large monorepo.
- Replayed real internal PR tasks using an eval suite (AuggieBench), running each task with and without AGENTS.md.
- Filtered for single-module scopes where agent docs could plausibly matter; compared against the “golden” PRs reviewed by senior engineers.

Key findings
- Same file, opposite effects: The very same AGENTS.md boosted best_practices by 25% on a routine bug fix but cut completeness by 30% on a complex feature task—because it triggered overexploration.
- Size matters: Best results came from a 100–150 line main AGENTS.md plus a few focused reference docs. Longer than that and gains reversed, especially in modules ~100 core files.
- Progressive disclosure wins: High-level guidance in the main file with tightly scoped references the agent can load on demand outperformed “cover everything here” docs.
- Write workflows, not essays: Numbered, step-by-step procedures moved agents from failing to finishing. Example: a 6-step deployment cut missing wiring files from 40% to 10%; correctness +25%, completeness +20%.
- Decide up front: Simple decision tables (e.g., React Query vs Zustand) improved adherence to conventions; best_practices +25%.
- Show, don’t tell: 3–10 line real code snippets/templates boosted code_reuse by ~20%. Too many examples led to mispatterning.
- Domain rules help—sparingly: Specific, enforceable rules (e.g., use Decimal for money) help; stacking dozens invites overexploration.
- Pair every “don’t” with a “do”: Prohibitions alone made agents cautious and meandering; pairing with concrete alternatives kept them moving.
- Modularity applies to docs too: Isolated submodules with modular docs performed best.

What failed
- Overlong, comprehensive AGENTS.md files.
- Walls of “don’ts” with no “dos.”
- Reference sections that pull the agent into reading sprees, spawning unnecessary abstractions and incomplete work.

Practical recipe
- Keep the main AGENTS.md ~100–150 lines.
- Include: 1–3 core workflows, 1–2 decision tables for common forks, a few real code snippets/templates, and a handful of specific, enforceable domain rules.
- Defer details to small, well-scoped reference files.
- For every “avoid X,” add “use Y.”

Bottom line: Treat AGENTS.md like a product surface. Short, procedural, and choice-constraining beats exhaustive and cautionary—and can deliver model-tier-sized gains without changing the model.

**The Verdict:** While the Hacker News community largely agrees that unified AI APIs and model routers solve real, painful problems for developers, Eden AI’s pitch as a “European OpenRouter alternative” was met with heavy skepticism due to legal transparency issues and questions surrounding actual data sovereignty. 

Here are the key takeaways from the discussion:

* **The Value of Convenience vs. Cost:** Users debated the merit of paying markups (with some noting surcharges of up to 55% depending on the model/provider) just for a proxy service. * **Missing Legal Transparency and "Imprints":** A massive red flag for the European developers in the thread was the absence of an “Imprint” (Impressum) on the website. In countries like Germany, Austria, and France, listing the legally responsible entity is a strict requirement for commercial websites. Though commenters dug into the terms and found it is run by a French company (Eden AI SAS), the lack of upfront transparency made users highly skeptical about trusting them for enterprise GDPR compliance and Data Processing Agreements (DPAs).
* **The "European" Label Misses the Mark:** Users were quick to point out that routing prompts through a European middleman achieves very little if the backend inevitably calls US or Chinese frontier models hosted on American cloud infrastructure (like AWS). Because these servers are subject to the US CLOUD Act, simply being an EU-based proxy does not guarantee data sovereignty. 
* **Favorable Comparisons to OpenRouter and Hugging Face:** The thread frequently compared Eden AI to its competitors. Developers complimented OpenRouter for being highly transparent about the specific data-retention policies for each provider on its platform—an area where Eden AI's documentation was found lacking. Additionally, several commenters pointed out that Hugging Face already offers a similar routing service (Inference Providers) and has an established European corporate presence.

**Bottom Line:** Developers love the idea of a “write once, route anywhere” AI gateway to avoid vendor lock-in and simplify billing. However, for a middleware company leaning heavily on its European identity as a selling point, the community feels Eden AI needs to substantially improve its legal transparency, DPA readiness, and data-retention clarity before overtaking the existing players.
