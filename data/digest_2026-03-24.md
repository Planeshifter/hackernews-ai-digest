SentrySearch: natural‑language search for dashcam footage using Gemini video embeddings

What it does
- Type a query like “red truck running a stop sign,” get back a trimmed clip from your dashcam footage.
- No transcripts or captions: it embeds raw video and text into the same vector space and matches them directly.

How it works
- Splits videos into overlapping chunks (defaults: 30s chunks, 5s overlap).
- Preprocesses locally (480p at 5fps by default) to keep uploads small and fast.
- Uses Google’s Gemini Embedding 2 to embed each video chunk; stores vectors in a local ChromaDB.
- On search, embeds your text query, finds nearest matches, and uses ffmpeg to auto‑trim the best clip from the source file.
- CLI flow: sentrysearch init (set Gemini API key) → sentrysearch index <directory> → sentrysearch search "<query>"; plus sentrysearch stats and verbose mode.

Cost and performance
- Author estimates ~$2.84 per hour of footage at defaults. Gemini processes video at 1 frame/second regardless of original FPS.
- Still‑frame skipping (on by default) can cut costs by avoiding calls for static chunks.
- Tuning trade‑offs: longer chunks and less overlap mean fewer API calls; preprocessing mainly speeds uploads and avoids timeouts, not billed frames.

Limitations
- Still‑frame detection is heuristic and can miss subtle motion.
- Events straddling chunk boundaries may be less accurately retrieved; smarter scene‑based chunking could help.
- Gemini Embedding 2 is in preview.

Why it matters
- Makes sub‑second semantic search over hours of dashcam or security footage practical with simple tooling and a local vector store.
- A clean, open‑source CLI that shows what native video‑text embeddings can do today.

Requirements and setup
- ffmpeg (or bundled imageio‑ffmpeg), Python package install, Gemini API key (via sentrysearch init or .env).
- Repo: https://github.com/ssrajadh/sentrysearch (open source; local DB, cloud embeddings).

### Hypura – A storage-tier-aware LLM inference scheduler for Apple Silicon

#### [Submission URL](https://github.com/t8/hypura) | 212 points | by [tatef](https://news.ycombinator.com/user?id=tatef) | [84 comments](https://news.ycombinator.com/item?id=47504695)

Hypura: Run oversized LLMs on Apple Silicon by streaming tensors across GPU/RAM/NVMe

- What it is: A Rust-based, storage-tier-aware inference scheduler for Apple Silicon that places model tensors across Metal GPU memory, RAM, and NVMe based on access patterns. Goal: make models that exceed physical memory actually run instead of swap-thrashing and crashing.

- Why it matters: Macs have fast unified memory and NVMe but limited capacity. Vanilla llama.cpp often OOMs on big models. Hypura turns that into “runs,” with no overhead when models fit in memory.

- How it works:
  - Hardware-aware placement after profiling your Mac (GPU working set, RAM, NVMe bandwidth).
  - GPU: attention, norms, embeddings. RAM: overflow via mmap. NVMe: on-demand direct I/O with prefetch (F_NOCACHE + pread).
  - Modes (auto-selected):
    - Full-resident: fits in GPU+RAM; zero overhead.
    - Expert-streaming (MoE like Mixtral): keep non-expert tensors on GPU; stream only the 2/8 experts selected per token; router interception + neuron cache hits 99.5% after warmup; speculative prefetch of likely experts.
    - Dense FFN-streaming (e.g., Llama 70B): keep attention+norms on GPU; stream dense FFN weights via a dynamically-sized pool buffer with auto-scaled prefetch depth.
  - No manual tuning: pool size, prefetch depth, and budgets derive from your machine.

- Performance (M1 Max, 32 GB, ~5.1 GB/s NVMe):
  - Qwen 2.5 14B Q4_K_M (8.4 GB): ~21 tok/s (fits; matches llama.cpp).
  - Mixtral 8x7B Q5_K_M (~31 GB): 2.2 tok/s; llama.cpp OOMs.
  - Llama 3.3 70B Q4_K_M (~40 GB): 0.3 tok/s; llama.cpp OOMs.
  - Key takeaway: When it fits, there’s no slowdown. When it doesn’t, Hypura trades a crash for usable throughput by streaming only what’s needed.

- Dev details:
  - Reads GGUF; vendors llama.cpp; uses Metal on Apple Silicon.
  - Builds with Rust 1.75+ and CMake. Homebrew tap coming.
  - Ollama-compatible HTTP API (/api/generate, /api/chat, /api/tags), so it slots into existing local AI tooling.

- Quick start:
  - git clone --recurse-submodules https://github.com/hypura/hypura.git
  - cd hypura && cargo build --release
  - hypura profile
  - hypura run ./model.gguf --prompt "Hello, world"
  - Tip: Start with --max-tokens 10 on new models.

Bottom line: Smart tiering + MoE-aware streaming makes big models runnable on 32 GB Macs, unlocking local LLMs that previously crashed, while staying zero-overhead for models that fit.

What they looked at
- PyPI as a public, consistently measured proxy for software output.

- No broad “2x more software” glut. The AI productivity bump doesn’t show up as more packages or a universal cadence lift.

- If you expected a visible, economy-wide output spike, PyPI doesn’t show it.

The author (@dnhkng) engaged actively, explaining the experiments utilized to validate the "thinking space." Using cosine similarity, they found that in the middle layers (the "reasoning" phase), representations of identical content across different languages (e.g., English vs. Mandarin) are nearly identical, whereas representations of different content within the same language diverge significantly. This suggests the model converts inputs into a universal, language-agnostic format for processing before decoding them back to the target language.

**The "RYS" Mechanism and Methodology**
Commenters discussed *why* duplicating layers (Repeat Your Self) works without retraining. The consensus—supported by the author—is that the middle layers form a coherent internal loop with compatible input/output distributions.

**Efficiency and Hardware Implications**
The discussion touched on the practical utility of RYS for local and edge inference:
-   **VRAM Usage:** While duplicating layers increases memory usage, the author suggested a promising inverse approach: deleting redundant layers to save VRAM, then duplicating highly efficient middle layers to recover the lost performance.
-   **Implementation:** Several users expressed interest in implementing this in inference engines like Llama.cpp, and the author mentioned ongoing work to integrate these finding with ExLlamaV3.

**Scientific Analogies**

Anthropic adds “auto mode” to Claude Code for safer unattended runs

Link: https://claude.com/blog/auto-mode

TL;DR: Claude Code now has an “auto mode” that automatically approves or blocks tool calls using a safety classifier, letting longer tasks run with fewer prompts. It’s meant to be a middle ground between per-action approvals and the risky --dangerously-skip-permissions.

How it works
- Before each file write or shell command, a classifier checks for risky patterns (mass deletes, data exfiltration, malicious exec).
- Safe actions run automatically; risky ones are blocked and Claude is redirected. Repeated blocks trigger a permission prompt.
- Expect fewer interruptions than default, but far less risk than skipping all permissions.

Why it matters
- Enables long-running workflows without babysitting.
- Reduces foot-gun risk versus fully bypassing permissions—useful outside throwaway sandboxes.

Caveats
- Not foolproof: ambiguous intent or missing context can still allow risky actions or block benign ones.
- Slight overhead in tokens/cost/latency.
- Still recommended for isolated environments.

Availability
- Research preview now for Team; rolling out to Enterprise and API “in the coming days.”
- Works with Claude Sonnet 4.6 and Opus 4.6.

How to try it
- CLI: claude --enable-auto-mode (then cycle modes with Shift+Tab).
- VS Code/Desktop: Settings -> Claude Code -> toggle Auto Mode, then select it in the permission mode dropdown.
- Admins: can disable via managed setting "disableAutoMode": "disable". Auto mode is off by default on Desktop; toggle via Organization Settings -> Claude Code.

Litellm PyPI supply‑chain attack: malicious 1.82.7/1.82.8 auto‑executes on Python start, steals secrets, tries K8s takeover; yanked after discovery

What happened
- On Mar 24, 2026 (10:52 UTC), litellm 1.82.8 was uploaded directly to PyPI (no matching GitHub tag), containing a litellm_init.pth file that runs on every Python interpreter start. Version 1.82.7 was also compromised.

- Hunt persistence:
  - Local: ~/.config/sysmon/sysmon.py and ~/.config/systemd/user/sysmon.service
