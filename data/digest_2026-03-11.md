TL;DR
- About half of AI PRs that pass SWE-bench Verified still wouldn’t clear maintainer review. Automated test pass rates overstate real-world merge readiness.

What they did
- Had 4 active maintainers from scikit-learn, Sphinx, and pytest review 296 AI-generated PRs that passed SWE-bench Verified, plus 47 “golden” human PRs previously merged.
- Reviewers were blinded to whether a PR was AI- or human-written and asked to accept or request changes, with reasons.

Key findings
- Maintainer merge rates are about 24 percentage points lower than the benchmark’s automated grader, after normalizing for reviewer noise using the golden baseline (68%).
- Apparent progress is slower under maintainer review: estimated improvement is 9.6 pp/year slower than automated scores (the authors call this suggestive, not definitive).
- Common rejection reasons: subtle functionality issues, breaking other code, code quality, and not following repo standards.

Why it matters
- Naively mapping benchmark scores to real-world usefulness likely overestimates how ready current agents are for production workflows.
- The gap seems less about fundamental capability limits and more about elicitation and process: unlike humans, agents in this setup got a single shot with no maintainer feedback loop.

Context and scope
- Covers 95 unique SWE-bench Verified tasks across 3 of 12 repos (about 19% of issues), spanning 5 LLMs from mid-2024 to mid/late-2025.
- Results don’t argue against benchmarks; they highlight that automated passes are necessary but not sufficient proxies for “mergeable” code.

### How we hacked McKinsey's AI platform

#### [Submission URL](https://codewall.ai/blog/how-we-hacked-mckinseys-ai-platform) | 458 points | by [mycroft_4221](https://news.ycombinator.com/user?id=mycroft_4221) | [190 comments](https://news.ycombinator.com/item?id=47333627)

How We Hacked McKinsey’s AI Platform — key points

- What happened: Researchers say an autonomous offensive agent, starting with only a domain and McKinsey’s responsible disclosure policy, found publicly exposed API docs for “Lilli,” the firm’s internal AI platform. Within two hours, it pivoted from an unauthenticated endpoint to full read/write access to the production database.

- Root cause: A classic-but-subtle SQL injection. An endpoint that logged user search queries safely parameterized values, but concatenated JSON key names directly into SQL. The agent detected reflected key names in error messages, used blind probing to map the query, and exfiltrated data. It then chained an IDOR to view cross-user activity. OWASP ZAP reportedly didn’t flag the issue.

- What was exposed (per the post):
  - 46.5M plaintext chat messages used for strategy, client work, and financials
  - 728k files (including ~192k PDFs, 93k Excel, 93k PowerPoint, 58k Word), with direct-download URLs
  - 57k user accounts, 384k AI assistants, 94k workspaces
  - 3.68M RAG chunks with S3 paths and internal metadata (proprietary research)
  - Model and prompt configs across 12 model types; flows through external AI APIs (including large OpenAI vector stores)

- The scarier bit: Prompt-layer compromise. Because system prompts lived in the same DB, write access meant an attacker could silently alter instructions—poison outputs, exfiltrate sensitive data via responses, strip guardrails—without code changes or obvious logs.

- Why it matters: Even well-resourced orgs can miss old-school vulns in new AI surfaces. Prompts and AI configs are now crown-jewel assets but rarely get code-like controls (auth, versioning, integrity checks).

- Takeaways for orgs:
  - Deny-by-default on all endpoints; don’t expose internal API docs
  - Treat JSON keys/field names as untrusted input; sanitize or whitelist schema
  - Add RLS/tenant isolation; encrypt sensitive content; use signed, expiring file URLs
  - Put prompts/configs under strict access control, versioning, and tamper detection
  - Red-team the entire AI pipeline (RAG stores, vector DBs, prompt stores), not just the app
  - Don’t rely solely on scanners; add custom tests for nonstandard injection paths

The post frames this as a responsible-disclosure research demo and a preview of how autonomous agents will expand the attack surface for AI-first systems.

Agent Browser Protocol (ABP): deterministic, stepwise web browsing for AI agents

What it is:
- A custom Chromium build with MCP and a simple REST API baked into the browser engine. No WebSockets, no CDP session wrangling—just HTTP.

Why it matters:

- Endpoints like POST /navigate, /click, /type return 200 OK with a screenshot + event log (e.g., tab_created, dialogs, file_chooser).
- Page freezes between steps so agents can reliably reason from the screenshot and events before proceeding.

- Cross-platform builds; TypeScript SDK available (theredsix/abp-npm). Docs include REST examples and manual install/compile guides.

Use cases:
- Reliable multi-step browsing for LLM agents, scraping, and UI testing where determinism and vision-based feedback are key.

**Agent Browser Protocol (ABP): Deterministic, stepwise web browsing for AI agents**

### Show HN: A context-aware permission guard for Claude Code

#### [Submission URL](https://github.com/manuelschipper/nah/) | 119 points | by [schipperai](https://news.ycombinator.com/user?id=schipperai) | [82 comments](https://news.ycombinator.com/item?id=47343927)

What it is
- A permission system that intercepts every tool call in Claude Code and decides based on what the command actually does, not just which tool it is.
- Uses fast, deterministic structural classification first; optionally falls back to an LLM for ambiguous cases.
- Every decision is logged and inspectable.

Why it matters
- Per-tool allow/deny doesn’t scale; “deny lists” are brittle, and “dangerously skip” modes can be catastrophic.
- “nah” aims to prevent accidental data exfiltration, destructive edits, and history rewrites while letting safe workflows run without friction.

How it works
- PreToolUse hook classifies action types (e.g., filesystem_read/write/delete, git_safe vs history rewrite, package runs, network ops).
- Context-aware: same command can be allowed in-project but prompted/blocked outside it.
- Path and content inspection: flags sensitive locations (~/.ssh, ~/.aws, .env) and secrets (e.g., private keys), blocks risky pipes (e.g., base64 -d | bash).
- Supports Bash, Read/Write/Edit, Glob, Grep, and generic MCP tools.

Notable examples
- git push — allowed; git push --force — prompt/block.
- rm -rf __pycache__ — fine; rm ~/.bashrc — blocked/prompted.
- Read ./src/app.py — allowed; Read ~/.ssh/id_rsa — blocked.
- Write config.yaml — allowed; curl ... | sh into ~/.bashrc — blocked.

Configuration
- Zero-config defaults; optional per-user (~/.config/nah/config.yaml) and per-project (.nah.yaml) policies.
- Action-type policies: allow, context, ask, block; 20 built-in types, profiles (full/minimal/none).
- Optional LLM layer (Ollama, OpenRouter, OpenAI, Anthropic, Snowflake Cortex) with caps so models can’t escalate beyond “ask.”

Getting started
- pip install nah; nah install (uninstall with nah uninstall && pip uninstall nah).
- In Claude Code, try /nah-demo for 25 live security cases (~5 minutes).

Bottom line
“nah” brings auditable, context-aware, default-sane guardrails to AI-driven tool use—aiming to stop the foot-guns (history rewrites, secret reads, dangerous pipes) without getting in your way for routine work.

### TADA: Speech generation through text-acoustic synchronization

#### [Submission URL](https://www.hume.ai/blog/opensource-tada) | 100 points | by [smusamashah](https://news.ycombinator.com/user?id=smusamashah) | [26 comments](https://news.ycombinator.com/item?id=47332054)

Hume AI open-sources TADA, a Text-Acoustic Dual Alignment approach that tightly syncs text and speech at a one-to-one rate. Instead of juggling long audio token streams or “semantic” intermediates, TADA emits one continuous acoustic vector per text token, keeping the LLM’s text and audio in lockstep. The result is a faster, lighter, and more reliable LLM-based TTS—aimed at on-device use without sacrificing expressiveness.

Why it matters
- Solves the core mismatch: audio carries far more info per second than text. TADA’s 1:1 alignment slashes context length and compute, and prevents skipped or hallucinated words by construction.

Key numbers
- Speed: Real-time factor 0.09 (~5x faster than comparable LLM TTS), operating at just 2–3 tokens/second vs 12.5–75 in other systems.
- Reliability: Zero hallucinations on 1,000+ LibriTTSR samples using CER > 0.15 as the threshold for failure.
- Quality: Human eval (EARS) scores 4.18/5 speaker similarity and 3.78/5 naturalness, ranking second overall among baselines.
- Context: A 2048-token window covers ~700 seconds with TADA vs ~70 seconds for typical approaches.

How it works
- Each LLM step yields both a text token and a single acoustic vector.
- A flow-matching head turns the LLM’s hidden state into acoustic features, decoded into audio and fed back in.
- The architecture’s strict alignment minimizes drift and content errors.

Use cases
- On-device voice (lower latency, better privacy), long-form narration and dialogue, and production settings needing low error rates.

Caveats
- Occasional long-form speaker drift (mitigated with online rejection sampling).
- “Modality gap” when generating text+speech together; Speech Free Guidance blends logits to help.
- Pre-trained for speech continuation; assistant-style use needs fine-tuning.
- English plus seven more languages for now.

Availability
- Open-source release with code, tokenizer/decoder, and 1B/3B Llama-based models, plus a live demo.
