- Built for hours‑to‑days runs with checkpointing, so interrupted jobs resume without starting over.
- Two ways to trigger: ask Claude to “create a workflow” or enable the new ultracode setting (effort xhigh; lets Claude decide when to use workflows). Best used with auto mode.

Use cases seen so far
- Repo‑wide bug hunts, performance and security audits with per‑finding verification.
- Large‑scale migrations and modernization (framework/API swaps, language ports spanning thousands of files).
- “High‑stakes” work double‑checked by independent and adversarial agents before surfacing.

Notable example
- Jarred Sumner used dynamic workflows to port Bun from Zig to Rust: ~750k lines over 11 days with 99.8% of tests passing. The pipeline mapped lifetimes, generated .rs files in parallel with two reviewers per file, then iterated fixes and later optimized copies via auto‑opened PRs. Anthropic notes this isn’t yet in production but was handled by workflows.

How it works under the hood
- Plan → parallel subtasks → independent angles + refuters → converge/verify → single coordinated result. Coordination happens outside the chat so plans don’t derail as scope grows.

Availability and cautions
- Research preview in Claude Code CLI, Desktop, and VS Code extension for Max, Team, and Enterprise (admin‑enabled) plans; also on the Claude API, Amazon Bedrock, Vertex AI, and Microsoft Foundry.
- Usage cost: substantially higher token consumption; first run shows a confirmation dialog. Orgs can disable via managed settings.

Getting started
- On by default for Max and Team; off by default for Enterprise (admin can enable).
- Turn on ultracode or ask Claude to create a workflow; use auto mode for best results.
- Docs and installers linked in the post.
