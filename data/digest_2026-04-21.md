### CrabTrap: An LLM-as-a-judge HTTP proxy to secure agents in production

#### [Submission URL](https://www.brex.com/crabtrap) | 120 points | by [pedrofranceschi](https://news.ycombinator.com/user?id=pedrofranceschi) | [51 comments](https://news.ycombinator.com/item?id=47850212)

CrabTrap: an open-source “LLM-as-a-judge” HTTP proxy for agent safety. It sits between your AI agent and the network, intercepting every outbound request and evaluating it against a policy. Decisions happen in real time, mixing static rules with an LLM judgment, and each allow/block is logged with whether it was a rule match or an LLM call. The team touts a 30-second setup and offers a live demo where you can edit rules and watch approvals/blocks. This targets a common production gap: centralized egress control and auditability for autonomous agents, independent of framework.

What’s not clear from the blurb: latency/cost when the LLM judge is invoked, the policy model/language, how TLS interception is handled, coverage beyond HTTP, and licensing/model support. Links to GitHub, a blog post, and a demo are provided. They’re also hiring.

### HN Digest: The "Who Watches the Watchmen?" Problem in AI Security
**Discussion Summary for CrabTrap (LLM-as-a-judge proxy)**

The Hacker News community met CrabTrap—an open-source HTTP proxy using an LLM to evaluate and block rogue AI agent requests—with a mix of architectural intrigue and heavy security skepticism. The thread quickly evolved from discussing the specific tool into a broader, highly spirited debate about the fundamental viability of "LLM-as-a-judge" security models. 

Here are the major themes from the discussion:

**1. The Prompt Injection Paradox**
The most prominent critique across the thread was the recursive vulnerability of using an LLM to secure an LLM. Multiple commenters pointed out that if an attacker successfully executes a prompt injection attack on the primary AI agent, they can easily format their payload to trick the *judge* LLM as well. One user noted that relying on an LLM to parse and block prompt injections is like putting a "subjective black box to protect a subjective black box." 

**2. Probabilistic vs. Deterministic Security**
Traditional security relies on deterministic rules (e.g., hardcoded ACLs, syntax checks). LLMs, inherently probabilistic, introduce a level of uncertainty that makes security purists uncomfortable. 
*   **The Skeptics:** Critics argued that adding an LLM layer is like adding "Swiss cheese" to a firewall, potentially giving developers a false sense of complete security. They highlighted that LLMs can still be bypassed by classic jailbreaks (like the "grandma's bedtime story" trick). 
*   **The Defenders:** Proponents countered that static rules alone cannot understand *intent*. They argued for "defense in depth," stating that while an LLM judge is not a silver bullet, layering it alongside deterministic checks provides a valuable additional net for catching complex, evasive behaviors. 

**3. Implementation & Architecture Debates**
Commenters offered several technical critiques and suggestions for how to safely implement this concept:
*   **Model Diversity:** A crucial point was raised regarding model families. If an agent runs on Claude, the judge should *not* be Claude, as they would share the exact same blind spots and vulnerabilities to specific prompt-injection patterns. 
*   **JSON Scoping:** There was a technical debate over whether embedding policy configurations inside rigidly scoped, deeply-nested JSON structures could effectively separate system instructions from user-generated payloads, or if LLMs are still too fragile at "syntax-in-syntax" parsing to reliably handle this without breaking.

**4. Community Tools & Alternatives**
As always, the HN crowd brought their own projects and references to the table. Tools like **EvalView** (for deterministic LLM drift detection) and **Pincer** (a highly optimized local model specifically for detecting prompt injections) were highlighted as complementary or alternative approaches. Additionally, projects with active bug bounties (like OpenClaw's $1,000 prompt-injection bounty) were referenced as proof of how difficult securing these systems truly is.

**The Takeaway:**
While developers love the idea of centralized, easily configurable egress control for AI agents, the HN consensus warns against treating an LLM-judge as a standalone security gate. Instead, it should be heavily sandboxed and treated as one non-deterministic layer in a much broader, fundamentally deterministic security stack.

GoModel: a Go-based, OpenAI-compatible AI gateway that unifies 10+ LLM providers

What it is
- A self-hosted gateway exposing OpenAI-style endpoints (/v1/chat/completions, /v1/responses, /v1/embeddings, /v1/files, /v1/batches) and routing them to multiple backends: OpenAI, Anthropic, Google Gemini, xAI (Grok), Groq, OpenRouter, Z.ai, Azure OpenAI, Oracle Generative AI, Ollama, and vLLM.

Why it matters
- One API for many models: swap vendors, add failover, or mix cloud and local models without changing client code.
- Drop-in usability: works with OpenAI-compatible SDKs; supports streaming and a passthrough mode for provider-specific features.

Notable features
- High-performance Go server with simple Docker/Compose deployment.
- Configure via env vars; register multiple instances of the same provider using suffixed variables (e.g., OPENAI_EAST_API_KEY).
- Observability built in: Prometheus metrics and docs included.
- Extras and edge cases covered: Oracle models can be enumerated via env when /models isn’t available; vLLM API keys optional; special Z.ai base URL for coding plan.

Getting started
- Run the container, pass only the provider keys you need, and hit /v1/chat/completions. The repo recommends loading secrets from an env file rather than -e flags to avoid leaks.

Who it’s for
- Teams standardizing on an OpenAI-style API while tapping multiple providers, or bringing local inference (Ollama/vLLM) under the same interface.

Here is a daily digest summary of the Hacker News discussion regarding **GoModel**, a Go-based, open-source AI gateway designed to unify 10+ LLM providers under the standard OpenAI API format.

### **The Abstraction Nightmare: Keeping Up with the AI Providers**
The most prominent theme in the thread was the sheer operational pain of maintaining LLM abstraction layers. Veteran proxy maintainers (including the creator of the popular `llm` CLI tool) noted that while the OpenAI Chat Completions format is the *de facto* standard, the ecosystem in 2025 is incredibly turbulent. 
*   **The moving targets:** Providers are constantly changing how they implement complex mechanisms like tool calling, structured outputs, and the new "reasoning" efforts. 
*   **The maintenance burden:** One commenter noted, "Implementing basic OpenAI/Anthropic protocols is trivial; building an AI gateway is a long journey of dealing with bugs, changes, and provider quirks."
*   **GoModel’s approach:** The author (`sntg-pl`) explained they manage this volatility by adhering to Postel’s Law (being lenient in what they accept/forward) and hitting REST APIs directly instead of relying on provider SDKs. For major upstream shifts, GoModel includes a passthrough mode. 

### **Security: Go Binaries vs. Python Supply Chains**
Several commenters juxtaposed GoModel against LiteLLM (a popular Python-based alternative). A major point of interest in GoModel relies on the language itself: Go. 
*   **Supply chain attacks:** The recent LiteLLM supply chain security incident was explicitly mentioned. Users expressed that a compiled, single-binary Go gateway inherently mitigates the runtime dependency risks associated with sprawling Python environments.
*   **Performance:** A developer looking to write their own Golang gateway noted they are currently benchmarking Rust vs. Golang, pointing out that compiled languages offer a much tighter grip on application security and performance for middleware.

### **Enterprise Governance, Quotas, and Security Constraints**
As teams try to scale LLM access internally, governance becomes a hurdle. Users brought up complex routing and credential scenarios:
*   Contributors asked how to manage user access, test environments, and quotas without generating direct provider keys for every team member.
*   The author highlighted GoModel's `User-Paths` concept, which allows admins to assign specific headers and track API usage per user/team without exposing root API keys. 
*   **Future Roadmaps:** In response to feedback, the author acknowledged the challenges of deep enterprise integration and noted that Vault integration (for secret storage) and Data Loss Prevention (DLP) tools are highly requested and on their radar.

### **A Mini-Showcase of Go-Based AI Tooling**
As is tradition on HN, the thread naturally evolved into a sharing ground for developers building AI tooling exclusively in Go. Notable projects dropped in the comments include:
*   **Sandboxing AI Agents:** Tools like `gbash` and `fence` were shared as robust, Go-based options for giving AI agents constrained bash / CLI environments without risking full system access.
*   **CLI workflows:** Other Go-based workflows like `hof` (leveraging Dagger for sandboxed AI code execution) and `shlly` (Shelley) were discussed by developers leaning into Go for AI engineering. 

**The Takeaway:** While the community remains slightly skeptical about the long-term viability of "universal AI APIs" due to vendors actively fragmenting their feature sets to lock users into their proprietary ecosystems, GoModel is being warmly received as a secure, fast, and necessary shield against the current chaos of the LLM API landscape.
