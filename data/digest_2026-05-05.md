Chrome is quietly dropping a 4 GB AI model on users’ machines, author says

- What’s new: Privacy researcher Alexander Hanff reports that recent Chrome versions auto-download an on‑device LLM (“Gemini Nano”) as weights.bin (~4 GB) into a folder named OptGuideOnDeviceModel, without an explicit consent prompt or a visible settings toggle. If users delete it, Chrome reportedly re-downloads it.

- How it was found: Beyond numerous Windows reports of disk space spikes, Hanff says he verified the behavior on a clean, automated macOS Chrome profile via the OS’s filesystem event logs (.fseventsd), showing the model arriving despite no UI interaction.

- Why it’s there: Google has marketed new Chrome AI features (e.g., “Help me write,” on‑device scam detection). The post claims these are enabled by default on eligible hardware and trigger the model download in the background.

- Can you stop it: The author says the only reliable ways are disabling Chrome’s AI features via chrome://flags or enterprise policy, or uninstalling Chrome. Deleting the file alone doesn’t stick because Chrome re-fetches it when eligible.

- Legal and environmental angle: Hanff argues the silent write breaches the EU ePrivacy Directive’s device access rule and GDPR principles (lawfulness, fairness, transparency, and data‑protection‑by‑design). He also estimates a one‑off global push could emit 6,000–60,000 tonnes CO2e given Chrome’s scale.

- Context: Follows Hanff’s earlier report that Claude Desktop registered a native‑messaging bridge across multiple Chromium browsers without consent. Both cases raise “reach across vendor trust boundaries” concerns.

- Open questions: Which exact Chrome versions/OSes are in scope, how defaults vary by region or rollout phase, and Google’s official rationale/opt‑out story. The post asserts the file is Gemini Nano; independent confirmation and a statement from Google are pending.

How to check on your machine
- Look for a folder named OptGuideOnDeviceModel (and a 4 GB weights.bin) under your Chrome user profile directory.
- If you don’t want it, disable Chrome’s AI/“Help me write”/on‑device features in chrome://flags or via enterprise policy; otherwise it may return after deletion.

### Accelerating Gemma 4: faster inference with multi-token prediction drafters

#### [Submission URL](https://blog.google/innovation-and-ai/technology/developers-tools/multi-token-prediction-gemma-4/) | 642 points | by [amrrs](https://news.ycombinator.com/user?id=amrrs) | [311 comments](https://news.ycombinator.com/item?id=48024540)

Accelerating Gemma 4 with Multi‑Token Prediction drafters

- What’s new: Google released Multi-Token Prediction (MTP) “drafters” for the Gemma 4 family, claiming up to 3x faster inference with no quality or reasoning loss. It’s open-source (Apache 2.0) and available now.

- How it works: A lightweight drafter proposes several next tokens while the heavier target model (e.g., Gemma 4 31B) verifies them in parallel. If the target agrees, the whole drafted chunk is accepted—and the target emits one extra token in the same pass. The drafter shares the target’s activations and KV cache to avoid redoing context work; edge models (E2B/E4B) get an extra boost from an efficient embedder clustering trick.

- Why it matters: 
  - Lower latency for chat, voice, and agent workflows
  - Faster local dev: runs 26B MoE and 31B dense on consumer GPUs with higher t/s
  - Better on‑device performance (and potential battery savings) for edge models
  - Claimed “zero quality degradation” since the target model retains final say

- Performance notes:
  - Reported tokens-per-second gains across LiteRT‑LM, MLX, Hugging Face Transformers, and vLLM
  - Mixture‑of‑Experts on Apple Silicon benefits from batching: batch sizes 4–8 yielded ~2.2x locally; similar batch gains noted on Nvidia A100
  - Real‑world speedups will depend on hardware, batch size, acceptance rate, and memory bandwidth

- Getting started: Weights are on Hugging Face and Kaggle; works with Transformers, MLX, vLLM, SGLang, Ollama, and can be tried on Google AI Edge Gallery (Android/iOS). Google posted a technical explainer covering architecture visuals, KV cache sharing, and the efficient embedders.

- The takeaway: Speculative decoding isn’t new, but tight drafter–target coupling (shared activations/KV) and edge‑oriented optimizations make this a notable, practical speedup for both cloud and on‑device Gemma 4 deployments—especially if you can batch.

**The Hacker News Reaction: Demystifying the "Magic" of Speculative Decoding**
The community response to Google's Multi-Token Prediction (MTP) drafters is highly enthusiastic, with users praising speculative decoding as an "amazingly clever invention" that feels almost too good to be true, given it speeds up inference with zero degradation in quality. 

Here are the central themes from the discussion:

*   **The "Aha!" Moment: Memory Bandwidth vs. Compute Bound:** 
    A large portion of the thread is dedicated to answering one unintuitive question: *How does running an extra model and doing more calculations result in faster times?* Commenters stepped in to explain that standard autoregressive LLM generation is **memory-bandwidth bound, not compute-bound**. Generating one token at a time means the GPU's arithmetic processing units sit idle while waiting for massive weight matrices to be moved from VRAM. MTP trades this idle compute capacity to solve the memory bottleneck. Once the heavy weights are loaded into fast memory, using them to verify several drafted tokens simultaneously is practically "free" in terms of time.
*   **A Classic CS Concept Reborn:** 
    Many developers immediately recognized the underlying philosophy, likening it to **CPU branch prediction**. Users pointed out that speculative execution is a foundational computer science concept (seen in multithreading, data prefetching, and database optimistic concurrency) that is now being successfully applied to Large Language Models.
*   **Intuitive Mental Models:** 
    Several users provided great plain-English examples of how this works. For instance, if an LLM outputs "The United," a smaller, cheaper draft model can easily guess "States of America." The main, heavier model then does a single parallel pass to verify those tokens. If the draft diverges (i.e., the main model wouldn't have chosen that path), the generation is simply discarded and re-calculated, keeping the target model in total control of the output quality. It holds massive potential for generating heavy code boilerplate where the next words are highly predictable.
*   **A Win for Local GPU Users:** 
    Commenters noted that MTP provides a similar hardware efficiency boost to batching. While cloud providers maximize hardware use by clustering multiple different user requests into a "batch," local developers running a single prompt usually can't do this. Speculative decoding effectively allows single-threaded, local users to get those "batching" efficiency benefits, greatly maximizing consumer GPU usage.

### Computer Use is 45x more expensive than structured APIs

#### [Submission URL](https://reflex.dev/blog/computer-use-is-45x-more-expensive-than-structured-apis/) | 461 points | by [palashawas](https://news.ycombinator.com/user?id=palashawas) | [258 comments](https://news.ycombinator.com/item?id=48024859)

Vision agents vs. API tools: the price tag, finally measured

What they did
- Same admin panel, same task, same model (Claude Sonnet), same dataset. Only the interface differed:
  - Path A: Vision agent via browser-use 0.12 (screenshots + clicks).
  - Path B: API agent via tool-use calling the app’s own event handlers (structured responses).
- Task: Find the “Smith” with the most orders, grab their most recent pending order, accept all their pending reviews, mark the order delivered (filters, pagination, cross-entity lookups, reads+writes).
- Code and run logs are open source.

Key results
- Out of the box:
  - API agent finished in 8 calls.
  - Vision agent failed (found 1 of 4 pending reviews; didn’t paginate because it had no signal to scroll).
- With a 14-step UI walkthrough prompt:
  - Vision agent completed the task, but took 14–22 minutes and 400k–750k input tokens per run.
- Aggregate metrics (mean ± sd):
  - Vision (Sonnet): 53 ± 13 steps; 1003s ± 254s; 550,976 ± 178,849 input tokens; 37,962 ± 10,850 output.
  - API (Sonnet): 8 ± 0 calls; 19.7s ± 2.8s; 12,151 ± 279 input; 934 ± 41 output.
  - API (Haiku): 8 ± 0 calls; 7.7s ± 0.5s; 9,478 ± 809 input; 819 ± 52 output.
- Variance:
  - Vision path varied wildly: 43–68 cycles, 749–1257s, 407k–751k input tokens across three runs.
  - API path was deterministic: identical 8 calls every time; tokens varied by ~±27.
- Haiku note: Couldn’t finish the vision path due to browser-use 0.12’s structured-output schema; on the API path it was the cheapest/fastest.

Why it happens (their take)
- Structural gap: An agent that must “see to act” will always pay a screenshot tax. Better vision reduces per-image errors, not the number of images needed to reach the data.
- Structured API responses surface facts directly (e.g., “page 1 of 4, 50 per page”) versus inferring pagination from pixels.
- The 14-step walkthrough is hidden engineering cost: without it, vision silently misses work; with it, you’re hand-authoring flows.

Implications
- Teams default to vision because building an MCP/REST surface per internal app is expensive. But if API surfaces can be generated from app logic/handlers, API agents win decisively on cost, speed, determinism, and reliability.
- For internal tools, exposing structured handlers flips the calculus: minutes and hundreds of thousands of tokens become seconds and ~10k tokens.
- Vision still fits for apps with no accessible structure, but it’s costly and brittle for routine internal workflows.

The 10 lessons
- Implement to learn: Writing code surfaces decisions specs miss.
- Rebuild often: Cheap code means frequent forks and rewrites to explore ideas.
- Invest in end-to-end tests: Test behavior, not implementation, to enable safe reinvention.
- Document intent: Preserve the why alongside code and tests to guide agents and humans.
- Keep specs in sync: Treat specs as living docs that evolve with the code.
- Find the hard stuff: The real value is in design, performance, security, resilience, architecture.
- Automate the easy: Turn repeatable wins into scripts, loops, and auto-reviews—without getting lost in tool-building.
- Develop your taste: When code is fast but feedback is slow, your judgment is the throttle.
- Agents amplify experience: Domain and stack intuition cut cycles and reduce aimless exploration.
- Code is cheap; maintenance isn’t: Support and security are “free as in puppies.”

Why it matters
- Bottlenecks shift from typing code to specifying intent, validating behavior, and making high-quality product decisions.
- Living specs + strong E2E tests let you refactor or regenerate fearlessly as models improve.
- The differentiator becomes taste and domain expertise, not raw coding speed.

Notable line: “Code is cheap, but maintenance, support, and security aren’t.”

A rigorous new benchmark called ProgramBench is putting top AI models to the ultimate test to see if they can architect and build entire software systems from scratch.
