### Why AI Sucks at Front End

#### [Submission URL](https://nerdy.dev/why-ai-sucks-at-front-end) | 101 points | by [tobr](https://news.ycombinator.com/user?id=tobr) | [144 comments](https://news.ycombinator.com/item?id=47738864)

TL;DR: LLMs are great at boilerplate and bad at bespoke UI. They speed up copy-paste work but stumble on precision, interaction, accessibility, and performance—especially as components get complex.

What it’s good at
- Scaffolding and wiring up well-worn patterns
- Mapping/migrating design tokens
- Generating feature outlines and lots of test boilerplate
- Quick snippets that get you 60–80% of the way on common UI

Where it falls down
- Pixel precision and custom interactions (e.g., scroll-driven animations, micro-interactions)
- Layout math, spacing, and combined component states
- Accessibility defaults (sprays ARIA without intent)
- Performance (defaults to heavy/janky approaches unless tightly constrained)
- Reliability declines as complexity rises; follow-ups often degrade quality

Why this happens
1) Trained on “ancient” patterns: The web is full of outdated solutions; modern CSS features are underrepresented.
2) It can’t render: An LLM isn’t a browser engine. Even with vision, it’s guessing at layout and intrinsic/extrinsic sizing.
3) No sense of “why”: It mimics code, not the architectural reasoning behind it (state models, a11y tradeoffs, performance budgets).
4) Zero environmental control: HTML/CSS run across unpredictable browsers, versions, viewports, inputs, and user prefs. The target is a moving kaleidoscope.

Big picture
- Use AI for commodity front-end work and as a rapid starting point.
- Expect to guide it with strict constraints, modern CSS primitives, and real-browser feedback loops.
- Treat outputs as drafts: you still need human judgment for a11y, performance, edge cases, and polish.

HN-angle takeaway: It’s impressive that LLMs can already produce “half-decent” UIs given these constraints, but the gap between template-level code and production-grade, accessible, high-performance front-ends remains stubbornly human.

  - Resumable, non-interactive workflows: continue a prior session and respond to queued prompts (e.g., select prompt 2).

  - ACP server: integrate with editors/clients (examples mention Toad and Zed) while still running a real Claude Code pane underneath.
  - Local-first via tmux; experimental xterm.js backend for headless fallback.
  - TypeScript library with runnable examples (code review, multi-session, session journaling).

- Not a replacement for the Claude SDK: You must authenticate via Claude Code, and all interactions run through real Claude Code sessions. It’s aimed at personal/ad‑hoc workflows, not OAuth or SDK-based apps.

  - One-off: npx @halfwhey/claudraband "review the staged diff"
  - Install: npm install -g @halfwhey/claudraband (use cband shorthand)
  - Local sessions: cband "audit the last commit"; cband sessions; cband continue <session-id> --select 2
  - Daemon: cband serve --host 127.0.0.1 --port 7842 then cband --connect localhost:7842 "start a migration plan"
  - ACP: cband acp --model opus (or swap models; can be driven by tools like Toad/Zed)

  - Requires Node.js or Bun and a pre-authenticated Claude Code.
