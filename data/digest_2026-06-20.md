Argusred v2.0.19: a self-serve CLI for code security scans with optional exploit verification

What it is
- A local, read-only security scanner for your repo. Pick modules (e.g., dependency vulns, secret/credential leaks, SQLi/XSS vectors, input validation, file perms), set agent permissions, and run. Output is a single markdown report with location, severity, cause, and fix direction.

Why it’s interesting
- Read-only by design: a Go harness blocks any write/exec, so scans won’t alter your code or run live attacks.
- Actionable results: you can optionally turn on Exploit Verification to safely reproduce findings so they’re confirmed, not just theoretical.
- Fast at scale: reported runs show ~10 min for ~30k LOC and ~40 min for ~1.5M LOC; modules execute in parallel.

How it works
- Install the CLI, cd into your repo, run argusred. A TUI lets you toggle modules and permissions.
- Exploit Verification modes:
  - Docker: attempts repro in an ephemeral, isolated container from your repo.
  - Live FS: runs against your checkout (still read-only).
  - Disabled (default): report-only, no repro.
- Report lands in .argusred/scan-<date>.md with an executive summary and per-finding details.

What it found (sample report)
- Critical issues in a demo app, including forgeable JWTs (shared public key + hardcoded private key), disabled signature verification in frontend auth, integer overflow bypass in transactions, SSRF/open redirect in OAuth flow, credentials in URL params, and hardcoded secrets.

Cost and account
- Free install. First run opens a Cosine sign-up and credits 2M free tokens. Ongoing scans bill under your Cosine account (same login as Cosine’s coding agent).

Scope and limits
- This is SAST-style analysis with optional safe repro. No fuzzing, DAST, or live exploitation in scan mode, and it won’t list findings it can’t ground in your code.

Gated Pen Test mode (separate)
- For authorized, active testing against live targets: port/service fingerprinting, TLS/header analysis, directory/endpoint enum, payload injection (SQLi/XSS/SSTI), with configurable aggressiveness. Produces an engagement summary with confirmed exploits. Access is gated via booking due to higher risk.

Quick start
- cd path/to/your/repo
- argusred
- Select modules, decide on exploit verification, run; read .argusred/scan-<date>.md

Bottom line
- A pragmatic, developer-friendly security pass that favors grounded, reproducible findings and a single, useful report—fast enough to run on real repos, with a clear upgrade path to a gated pentest when you need active probing.

### LLMs Are Complicated Now

#### [Submission URL](https://ianbarber.blog/2026/06/19/llms-are-complicated-now/) | 199 points | by [matt_d](https://news.ycombinator.com/user?id=matt_d) | [67 comments](https://news.ycombinator.com/item?id=48605355)

TL;DR: The clean, stack-of-Transformers era is over. Modern open models like Llama 3 vs. Nemotron 3 Ultra showcase an explosion of architectural tricks—attention variants, MoE-style routing everywhere, multimodal mixing, and multi-GPU inference—pushing LLMs toward recsys-level complexity. The only sustainable way to iterate is to design for composability and verifiability up front.

Highlights
- Attention isn’t one thing anymore: query grouping, compressed, sparse, linear, sliding-window, etc., each with different performance profiles.
- Routing has escaped MoE feed-forwards: researchers now route attention blocks and even residual streams.
- Multimodality has gone from bolt-on encoders to deeply mixed-in components.
- Inference spans multiple GPUs, inserting communication ops that further segment the graph.
- Parallels to recommendation systems: once “simple two-tower” models, recsys became complex because performance optimizations turned from nice-to-have into load-bearing necessities.
- Auto-kernel “agents” won’t save you without a fixed, trustworthy baseline. If variant B is 10x slower than optimized A, you can’t even evaluate it meaningfully.
- FlexAttention in PyTorch is a positive pattern: a composable, Triton-templated way to explore whole classes of attention ops with minimal perf penalty, enabling faster, safer research loops.
- Pointed note: Andrej Karpathy’s move to Anthropic aligns with building richer auto-research loops—but architectural minimalism and composability matter just as much as agentic automation.

Why it matters
- Research velocity now hinges on infra that’s both fast and flexible. Without composable, partially fused baselines, exploration stalls.
- The frontier is shifting from “optimize this known model” to “safely explore many model families,” which demands kernel/template systems designed for verification and swap-ability.

Here is a summary of the Hacker News discussion surrounding Ian’s blog post on the escalating complexity of LLMs:

**1. The "Bitter Lesson" and the S-Curve of Engineering**
A major part of the discussion centered around Rich Sutton’s famous "Bitter Lesson" (the idea that raw computation and scaling always eventually beat clever, human-engineered heuristics). However, commenters noted a cyclical "engineering lifecycle." When compute and data provide massive gains, engineering complexity takes a back seat. But as those scaling gains naturally shallow out (hitting the top of an S-curve), developers are forced to invest heavily back into complex, domain-specific engineering and architectural tweaks to squeeze out smaller incremental gains and better hardware utilization. 

**2. The Open-Source Implementation Nightmare**
The theoretical complexity mentioned in the article is having very real, painful consequences for open-source maintainers. Commenters pointed to projects like `llama.cpp`. In the past, implementing a new model was straightforward. Today, adding support for models like DeepSeek v3, GLM 5.2, or Kimi is becoming incredibly difficult due to the myriad of customized attention implementations, complex MoE routing, and unique architectural features, resulting in partial or delayed implementations. 

**3. Critiques of the Author's Framing**
Some users pushed back on the article's core comparison. They argued that contrasting a standard attention model like Llama 3 against highly specialized, multi-family architectures like GLM 5.2 or Nemotron was "cherry-picking" or an apples-to-oranges comparison meant to exaggerate the shift. However, even critics generally agreed that the underlying premise holds true: pushing for total hardware efficiency is turning modern ML into a trial-and-error empirical mess, heavily reliant on complex hacks. One user compared the current state of Machine Learning to physics right before the discoveries of relativity and quantum mechanics—effective, but fundamentally an "iterative mess" that no one fully understands from first principles.

**4. The "Written by AI" Meta-Debate**
A large, highly contentious portion of the thread was entirely derailed by accusations that the original blog post was written by an LLM (specifically Claude). 
*   **The Detectives:** Some users claimed the article featured classic "Claudish" tells, pointing to specific vocabulary (like "load-bearing"), formatted lists, and sentence structures. 
*   **The Pushback:** Other users, including professional writers, fiercely defended the text, pointing out distinct human quirks (like specific timeline phrasings and grammatical structures) that LLMs tend to iron out. 
*   **The Fatigue:** The derailment caused significant frustration. One user proposed a "Scrabble rule for HN AI challenges": if you accuse an article or comment of being AI-generated and are proven wrong, you should receive a massive penalty, as these low-effort accusations frequently ruin high-quality technical discussions.

### Temporary Cloudflare accounts for AI agents

#### [Submission URL](https://blog.cloudflare.com/temporary-accounts/) | 233 points | by [farhadhf](https://news.ycombinator.com/user?id=farhadhf) | [137 comments](https://news.ycombinator.com/item?id=48608394)

Cloudflare ships “Temporary Accounts” so AI agents can deploy without sign-ups

- What’s new: Agents can now run wrangler deploy --temporary to push a Cloudflare Worker instantly, no OAuth, dashboard clicks, or MFA. Cloudflare spins up a throwaway account, hands Wrangler an API token, and returns a claim URL.
- How it works: If Wrangler isn’t authenticated, it now hints the new --temporary flag. Deployments stay live for 60 minutes. You (or your agent) can iterate and redeploy freely during that window. Click the claim link anytime to convert the temp account into a permanent one—Workers plus any provisioned resources (databases, bindings) come along.
- Why it matters: Removes the “human-in-the-loop” auth wall that stalls background agents. Supports the rapid write → deploy → verify loop agents rely on, and avoids forcing users to pre-create accounts on third-party services.
- Bigger picture: Follows Cloudflare’s recent Stripe-backed protocol for agent-led provisioning (accounts, subscriptions, domains, API tokens) and WorkOS’s auth.md effort to standardize agent sign-ups.
- Fine print: Temp accounts auto-delete after 60 minutes if unclaimed; capabilities may evolve—check docs for limits.
