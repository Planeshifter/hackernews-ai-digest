The post takes aim at RTK, a popular tool that compresses terminal output for LLM agents, arguing its headline claim—“same intelligence at 1/10th the price”—glosses over real costs and real risks.

Key points:
- Savings aren’t your bill: The touted 60–90% reduction is on CLI output only, not on the biggest cost drivers like file/repo context, system prompts, or the model’s own reasoning tokens. The author calls features like rtk gain more screenshot bait than true cost optimization.
- Silent failure risk: Reports of mangled or dropped lines mean agents may miss critical stack traces without knowing it. Because the model isn’t aware of the compression, errors can go undetected and derail tasks.
- Missing the only metric that matters: There are lots of “tokens saved” charts, but no task success benchmarks (e.g., SWE-bench). If compression harms accuracy, agents can loop or hallucinate and end up burning more tokens.
- A feature, not a platform: This kind of output compaction could be a native flag on mainstream tools (e.g., --compact, --json-stream). If CLIs adopt it directly, RTK’s edge evaporates.
- Brittle against CLI churn: Heavily regex-based parsing of human-readable stdout/stderr is fragile. Minor format changes in git/npm/cargo could break filters and reintroduce the “silent failure” problem.

Takeaway: The author warns that RTK trades reliability and semantic completeness for a flashy token reduction. Until it ships hard accuracy benchmarks and robust failure detection, placing it in a production agent’s critical path looks like an operational risk that isn’t worth the discount.

- What it is: A Rust-powered site auditor that crawls sites for broken links, redirects, missing metadata, and 40+ checks spanning technical SEO and “GEO” (AI-search readiness). It returns two scores per run: Health (SEO) and GEO (how citable/structured your content is for generative engines).

- Why it’s notable: Ships not just a CLI but also an MCP (Model Context Protocol) server, so agents like Claude Desktop/Code, Cursor, or Cline can autonomously run full audits and act on results. It also includes “skills” (playbooks) that let agents run audits via npx even if the binary isn’t preinstalled.

- Key features:
  - Respects robots.txt, seeds from sitemap.xml; supports include/exclude globs, max depth/pages, and concurrency.
  - Outputs: json (default, machine-readable), pretty (terminal), csv (issues), html (shareable report).
  - CI-friendly: --fail-on error|warning for gating; severity filtering; saves report history.
  - explain_issue tells you why a finding matters and how to fix it.
  - Tools exposed to agents: crawl_site, audit_url(s), explain_issue, list_rules, list/get_reports.

- Install and run:
  - CLI + MCP via npm (auto-installs native binary): npm i -g crawlie
  - macOS desktop app: signed .dmg in Releases
  - From source: Rust (cargo) for engine/CLI/MCP; pnpm+Node for desktop app
  - Quickstart:
    - Crawl a site: crawlie crawl https://example.com --format pretty
    - Audit a page: crawlie audit https://example.com/pricing
    - HTML report: crawlie crawl https://example.com --format html -o report.html
    - JSON for scripting: crawlie crawl https://example.com --format json -o report.json

- Agent integration:
  - MCP server on PATH after npm i -g crawlie; add to Claude Desktop/Code or any MCP client (JSON-RPC over stdio).
  - One-step Claude Code plugin: claude plugin marketplace add spronta/crawlie; claude plugin install crawlie@spronta
  - Skills auto-run crawlie via npx if missing.

- Use cases: Pre-launch QA (4xx/5xx, redirects, broken links), GEO optimization (structured data, semantic HTML, answer-ready content, authorship/E-E-A-T), and automated agent workflows.

Repo: https://github.com/spronta/crawlie

### Migrate from OpenClaw

#### [Submission URL](https://hermes-agent.nousresearch.com/docs/guides/migrate-from-openclaw) | 118 points | by [JumpCrisscross](https://news.ycombinator.com/user?id=JumpCrisscross) | [98 comments](https://news.ycombinator.com/item?id=48586005)

Hermes ships a one-command migrator for OpenClaw (plus legacy Clawdbot/Moldbot) setups

What’s new
- Run hermes claw migrate to import your OpenClaw persona, memories, skills, models/providers, agent behavior, session reset rules, MCP servers, and TTS settings into Hermes. It auto-detects ~/.openclaw/, ~/.clawdbot/, and ~/.moltbot/ (and old config filenames).
- Always shows a full preview before applying. Creates a zip snapshot of ~/.hermes/ for easy rollback unless you pass --no-backup.
- If your OpenClaw was multi-provider, hermes setup --portal can collapse it to a single OAuth via the Nous Portal (300+ models plus Tool Gateway) after migration.

Quick start
- Preview then migrate: hermes claw migrate
- Preview only: hermes claw migrate --dry-run
- Full apply, include secrets, no prompts: hermes claw migrate --preset full --migrate-secrets --yes

Key options
- --preset full | user-data (presets never import secrets)
- --migrate-secrets (puts provider keys into ~/.hermes/.env)
- --overwrite (apply even on file conflicts)
- --skill-conflict skip | overwrite | rename
- --workspace-target <path> (where AGENTS.md lands)
- --source <path> (custom OpenClaw dir)
- --yes (skip confirmation), --no-backup

What gets migrated
- Persona and instructions: SOUL.md and AGENTS.md (AGENTS.md requires --workspace-target)
- Memory: MEMORY.md and USER.md merged/deduped; daily memory files folded into main (entries parsed with § delimiter)
- Skills from four places (workspace, shared, personal, project-level) into ~/.hermes/skills/openclaw-imports/ with conflict handling via --skill-conflict
- Models/providers: default model and custom providers mapped (supports short and hyphenated provider types); API keys only if --migrate-secrets → ~/.hermes/.env
- Agent behavior: maps max turns (timeoutSeconds/10, capped 200), verbose, reasoning_effort (high/medium/low mapping), compression on/off and summary model, human delay mode/timings, timezone, terminal timeouts, Docker sandbox backend/image
- Session reset policies: daily/idle modes, at_hour, idle_minutes; falls back to resetTriggers if needed
- MCP servers: command/args/env/cwd for stdio, url for HTTP/SSE, tool include/exclude
- TTS: reads from canonical and legacy OpenClaw locations; maps provider to tts.provider and carries provider-specific fields (e.g., ElevenLabs voice ID) into Hermes config

Why it matters
- Turns scattered OpenClaw state into a single Hermes config with safety rails (preview + backup), and sets you up to consolidate providers under one OAuth with access to hundreds of models and tools.

Post-migration checklist
- Verify AGENTS.md is in your chosen --workspace-target
- Confirm default model/provider mappings and that keys loaded from ~/.hermes/.env (only if you passed --migrate-secrets)
- Check skills for desired conflict resolution (skip/overwrite/rename)
- Test MCP server connections and tool filters
- Sanity-check session reset behavior, terminal timeouts, compression settings, and TTS voice selection

Most commenters who tried Hermes agreed that it feels significantly more polished, focused, and stable compared to OpenClaw. However, some power users felt Hermes still suffers from "skill bloat." **The "Agent Use-Case" Debate: Utility vs. Slop**
A fierce debate broke out over what these agents are actually useful for. Some users shared success stories of using OpenClaw for automated marketing pipelines (scoring leads and messaging via Slack). Critics quickly pushed back, dismissing these automated lead-generation use cases as "marketing slop" and "spam." Several commenters argued that assigning "personalities" to AI tools misses the point; they don't want a tamagotchi or a digital doll, they just want a programmable context window to definitively solve tasks. 

**The Minimalist Rebellion**

**Bottom Line:** The community is glad Hermes built a safety rail/migration path away from OpenClaw's messy state, but there is a growing backlash against heavy "all-in-one" AI agent frameworks in favor of lightweight, custom-built scripts.

In today’s top discussion, the Hacker News community is debating the boundaries between employee civic advocacy, corporate loyalty, and brand representation. A major part of the discussion revolves around Seattle’s local ordinances, which prohibit employers from discriminating based on political ideology. **The Naming Dispute: "Amazon Employees for Climate Justice"**
The most heated debate centers on the activist group's name. 
*   **Defenders of the engineers** argue that the group's name is factually accurate—they *are* Amazon employees. They argue that identifying oneself as an employee does not automatically mean one is claiming to be an official spokesperson for the corporation, and that the engineers made it clear they were speaking in a personal capacity. 
*   **Critics of the engineers** argue that incorporating Amazon's trademarked name into an activist group's title eliminates any "wiggle room." They view this as "leaching" off the company's brand for political clout and intentionally blurring the lines of representation, which they believe fully justifies HR's intervention. 

**Conflict of Interest and Sabotage**
Beyond the legalities, several commenters debated the ethics of the employees' actions. Some users argued that it is inherently hypocritical to accept a paycheck from an employer while actively "throwing wrenches" into its $200 billion infrastructure plans. One commenter compared the situation to a doctor or nurse actively campaigning to block the construction of a new hospital wing they personally disagree with. Conversely, others noted that tech workers shouldn't be expected to surrender their local political voice or environmental concerns simply because their employer is tied to the issue. 

**The Takeaway**
The thread highlights a growing tension in the tech industry. As massive AI data center buildouts face increasing friction from local municipalities over power grids and environmental concerns, companies and their workforces are finding themselves at odds. The discussion underscores a difficult question for the modern tech landscape: where does a citizen's right to local political activism end, and an employee's fiduciary duty to not sabotage their employer's business strategy begin?
