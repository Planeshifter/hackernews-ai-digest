llmfit: Find the best LLMs that actually run on your machine

What it is
- A terminal tool that detects your CPU/GPU/RAM and ranks “hundreds of models” by how well they’ll run locally. It scores each model on quality, speed (tok/s estimate), memory fit, and context length, then recommends the right quantization and run mode (GPU, CPU-offload, CPU-only).
- Comes with an interactive TUI by default plus a classic CLI. Works with local runtimes like Ollama, llama.cpp, and Apple MLX, and supports multi-GPU and MoE models.

Why it matters
- Saves hours of trial-and-error when picking model sizes and quants for your hardware.
- “Plan mode” flips the workflow: set a target model/context/throughput and it estimates the VRAM/RAM/CPU you’d need, including upgrade deltas.

Notable features
- Live hardware detection; provider-aware downloads; filters for fit/availability; installed-first view.
- Dynamic quantization selection and speed estimation per model.
- GPU VRAM override when autodetect fails: e.g., llmfit --memory=24G.
- Themes, search, sorting by score/params/mem%/context/date/use case.

Quick install
- macOS/Linux (script): curl -fsSL https://llmfit.axjns.dev/install.sh | sh
- Homebrew: brew install llmfit
- Cargo: cargo install llmfit
- Source: git clone https://github.com/AlexsJones/llmfit.git && cd llmfit && cargo build --release

Quick use
- TUI: llmfit
- Top recs (JSON): llmfit recommend --json --limit 5
- Plan hardware for a config: llmfit plan "Qwen/Qwen3-4B-MLX-4bit" --context 8192 --target-tps 25

Caveats
- Autodetection can fail on some systems; use --memory to override.
- Throughput/memory are estimates, not guarantees.

What it is

What’s inside the one file

Why it matters
- A compact, readable reference that demystifies LLMs: you can follow every moving part without frameworks.
- Culminates a lineage of minimal projects (micrograd, makemore, nanoGPT) focused on clarity over performance.

Caveats
- Educational, not production: it’s pure Python, character‑level, and optimized for simplicity, not speed or scale.

TL;DR: An interactive, from-first-principles tour of decision trees. Solid primer with excellent visuals.

What’s inside
- Intuition first: Step-by-step axis-aligned splits turn simple if–then rules into piecewise regions; you see how a few thresholds carve up the plane.
- Don’t go too deep: Demonstrates how additional splits start fitting noise; ties directly to the bias–variance tradeoff and why we stop early or prune.
- How splits are chosen: Entropy as an impurity measure; information gain selects the split that most reduces uncertainty. Introduces the greedy, top-down ID3 procedure.
- Beyond classification: Notes trees handle both classification and regression; leaves output a class or a numeric value.
- Interactives: Play with points to feel impurity change, and watch the tree update as you move thresholds. Great for building intuition.

Why it matters
- Decision trees are interpretable, fast baselines that handle nonlinearity and mixed features with minimal preprocessing.
- They’re the backbone of Random Forests and Gradient-Boosted Trees—today’s tabular-workhorses—so grokking splits/impurity pays off.

Practical takeaways
- Control complexity: max depth, min samples per split/leaf, and pruning (or cost-complexity) tame overfitting; validate with cross-validation.
- Impurity options: Entropy vs. Gini (often similar); gain ratio can reduce bias toward many-valued categorical features.
- Know the limits: Axis-aligned splits can be brittle/unstable, trees don’t extrapolate well in regression, and small data can lead to high variance—ensembles fix much of this.

Bottom line: A crisp, hands-on explainer that makes “nested rules” feel powerful—and shows exactly how trees decide where to cut. Great for newcomers; a tidy refresher for veterans.
