### MegaTrain: Full Precision Training of 100B+ Parameter LLMs on a Single GPU

#### [Submission URL](https://arxiv.org/abs/2604.05091) | 321 points | by [chrsw](https://news.ycombinator.com/user?id=chrsw) | [56 comments](https://news.ycombinator.com/item?id=47689174)

MegaTrain: training 100B+ LLMs at full precision on a single GPU by streaming weights from CPU RAM

- The pitch: MegaTrain flips the usual GPU-centric model on its head. It keeps model parameters and optimizer states in host (CPU) memory and treats the GPU as a stateless compute engine. For each layer, it streams weights in, computes, and streams gradients back out—minimizing what must persist on the device.

- How it works
  - Memory-centric design: Parameters and optimizer states live in system RAM; the GPU only holds what’s needed for the current layer.
  - Bandwidth-aware pipeline: Double-buffered execution overlaps three phases—parameter prefetch, compute, gradient offload—across multiple CUDA streams for near-continuous GPU utilization.
  - Stateless layer templates: Replaces persistent autograd graphs with templates that bind weights on the fly as they arrive, cutting graph metadata overhead and allowing flexible scheduling.

- Results (from the paper)
  - Single H200 GPU with 1.5 TB host memory: reliably trains models up to 120B parameters (full precision).
  - Throughput: 1.84× faster than DeepSpeed ZeRO-3 with CPU offloading when training a 14B model.
  - Long context: Enables training a 7B model with 512k tokens on a single GH200.

- Why it matters
  - Trains models well beyond VRAM limits without multi-GPU sharding or quantized training.
  - Makes very-long-context training feasible on a single high-end node.
  - Challenges the assumption that you need massive GPU clusters for 100B+ full-precision training—if you have ample RAM and fast interconnect.

- Caveats to note
  - “Single GPU” still assumes a server with 1.5 TB system memory and high CPU–GPU bandwidth; that’s specialized (e.g., GH200/Grace Hopper-class setups).
  - Performance hinges on efficient overlap of streaming and compute; CPU–GPU bandwidth remains the fundamental bottleneck.
  - Claims are from the paper; broader benchmarks and open-source availability weren’t detailed in the abstract.
