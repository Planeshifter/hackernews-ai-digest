  - It’s specialized (single-shot tool calls) and the authors note small models can be finicky.

### Show HN: Statewright – Visual state machines that make AI agents reliable

#### [Submission URL](https://github.com/statewright/statewright) | 109 points | by [azurewraith](https://news.ycombinator.com/user?id=azurewraith) | [50 comments](https://news.ycombinator.com/item?id=48108778)

What it is
- An open-source Rust engine plus MCP plugin that enforces state-machine workflows for AI coding agents. In each phase (planning, implementing, testing), only specific tools and commands are available; attempts outside the allowed set are rejected with guidance on how to proceed.
- Works across Claude Code, Cursor, Codex, opencode, and Pi so you define a workflow once and get consistent guardrails everywhere.

Why it matters
- Instead of “bigger model, longer prompt,” it narrows the problem so the agent focuses and stops flailing. It breaks common failure modes like read-loop spirals and risky shell usage, and it can lift smaller local models to reliably complete tasks they otherwise fail.

How it works
- Deterministic Rust core evaluates states, transitions, and guards (no LLM in the loop).
- MCP plugin enforces per-state tool visibility and command allow-lists.
- Typical flow: planning (read-only tools), implementing (edits + tightly scoped shell, destructive ops and redirects blocked), testing (designated test commands), completion.
- Guardrails include: per-state tool enforcement; bash discernment (blocks >>, rm, shred, interpreters in the wrong phases); edit caps (max lines/files); conditional transitions; human approval gates; env scoping; per-session isolation.

Evidence
- On a 5-task SWE-bench subset with local models, two mid-size models improved from 2/10 to 10/10 when run under Statewright constraints, same tasks/hardware. Below ~13GB models could call tools but couldn’t retain enough file content to edit accurately (claimed as a model limit, not a Statewright one).
- Frontier models already avoid obvious catastrophes, but the structural win was fewer loops and tighter reasoning.

Try it
- In Claude Code (free tier): “/plugin marketplace add statewright/statewright”, then “/plugin install statewright”, then “/statewright start bugfix”. You’ll be prompted for an API key from statewright.ai.

Caveats
- Reported results are on a small subset (5 tasks); one model needed a specialized edit_line tool; another was tested on 2/5 tasks.

Here is a summary of the Hacker News discussion regarding Statewright:

**The Licensing & Patent Debate**
The most prominent topic in the thread centered on the project’s dual-licensing and mention of a provisional patent. Users expressed hesitation, noting that combining open-source claims with patents often makes developers legally wary to adopt or contribute. 
The creator clarified the structure: the core Rust engine is fully open-source (Apache 2.0), while the plugins are released under the Fair Source License (FSL 1.1), which restricts hosted competition for three years before converting to open-source. The patent is intended defensively against large tech companies attempting to scoop a managed service, not against individual developers. Prompted by the pushback, the creator actively shipped changes during the discussion, adding explicit patent grants for solo developers/researchers to the repo's FSL template and fixing conflicting license tags in the `Cargo.toml` file.

**Validation of the Architecture**
Several developers chimed in to validate the state-machine approach, noting that they have built similar custom harnesses. The consensus is that providing smaller local LLMs with massive context windows and unlimited tool access often degrades performance. By rigidly constraining which tools are visible or executable based on the current state (Plan, Build, Test), developers are successfully forcing models to work methodically rather than getting caught in execution "death spirals." 

**Prompt Caching vs. State Transitions**
A highly technical question was raised regarding API economics: if the tool list changes during every state transition, wouldn't that invalidate LLM prompt caching and heavily increase API costs and latency? 
The creator explained that there are two ways to handle this. In the current Claude Code MCP plugin, the tool schema remains static (keeping the cache intact), but the Statewright engine blocks disallowed tools at *execution time* with an error message. In their local/research setups, the tool schema is genuinely swapped per state, which breaks the cache but significantly reduces the overall token count and decision complexity for the model. 

**General Frustrations with Agent Tooling**
Users vented about the current state of mainstream coding agents, particularly criticizing UIs that hide actual operations behind slow animations or utilize inefficient underlying tools. Commenters welcomed tools like Statewright that add visibility, determinism, and tighter guardrails to the development loop. The creator was highly engaged throughout, even releasing UI label updates to the project's visualizer based on real-time feedback in the thread.
