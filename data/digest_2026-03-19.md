### NanoGPT Slowrun: 10x Data Efficiency with Infinite Compute

#### [Submission URL](https://qlabs.sh/10x) | 162 points | by [sdpmas](https://news.ycombinator.com/user?id=sdpmas) | [43 comments](https://news.ycombinator.com/item?id=47444072)

NanoGPT Slowrun claims 10x data efficiency by swapping “more data” for “more compute” via ensembles, heavy regularization, looping, and targeted arch tweaks

- The headline: An ensemble of 1.8B-parameter base models (18B total) trained on just 100M tokens reportedly matches a standard LM trained on ~1B tokens. Earlier ablations showed 3.8x and 8x; the team now claims 10x within weeks.

- Why this is interesting: If compute keeps growing faster than high-quality data, scaling laws that demand proportional data will hit a data bottleneck. This work argues you can keep improving by spending compute (ensembling/overparameterization) while holding data fixed.

- Chinchilla defied: For 100M tokens, Chinchilla-style guidance suggests ~5M parameters. Slowrun instead runs 1.8B per model and 18B in aggregate—about 3600x the “optimal” total size.

- Ensembles flip overfitting dynamics: Training individual models longer worsened their own loss (3.295 → 3.310 from 12 → 18 epochs), but improved ensemble loss (3.185 → 3.166). Longer training makes base models diversify in ways the ensemble can exploit.

- Chain knowledge distillation: Train models sequentially; each new model distills from the immediately previous one (α=0.5, T=1). Only the last model is kept as teacher, so memory stays constant. With 8 chained models, individual loss plateaus ~3.20 but ensemble loss drops to 3.126 (moving from ~7x to ~8x data efficiency). They see “headroom” from scaling K further.

- Heavy regularization because heavy overparameterization: L2 weight decay up to 1.6 and dropout 0.1—far above common practice—work well on 100M tokens when training 1.8B-parameter models. This aligns with findings that optimal weight decay grows in data-constrained regimes.

- Looped transformers: After mid-training, they re-run a middle block (layers 15–24) four times per token, improving inductive bias by spending more compute per prediction. Don’t loop the final layers; that hurt.

- Architecture tweaks that stacked: 
  - Exclusive Self Attention (remove self-value projection)
  - EMA of weights with tuned weight decay
  - Half-truncated RoPE; partial key offset for single-layer induction heads; tuned residual lambdas
  - U-Net-style skip connections between mirrored layers
  - SwiGLU activations
  - Value embeddings via a learned projection from input embeddings

- What to watch:
  - Inference cost scales with number of models; distilling the ensemble back into a single model would be the pragmatic next step.
  - Results are reported mostly in validation loss; broader benchmark wins and robustness would strengthen the claim.
  - The looping and arch-search heuristics need formalization to transfer cleanly to larger scales.

Bottom line: They argue you can meaningfully trade compute for data during pretraining. If these ensemble/distillation + regularization + looping tricks generalize, data-limited orgs could push much further without scraping 10x more tokens.

This manifesto argues that as AI coding agents pump out more code, teams need stronger conventions to prevent rapid “sloppification.” It proposes a clear split between semantic and pragmatic functions, stricter modeling to make invalid states unrepresentable, and type branding to stop subtle ID mix-ups—packaged as a reusable “skill” you can give to agents (npx skills add theswerd/aicode) and plug into Cursor.

Key ideas
- Code should be self-documenting. Prefer small, explicit “semantic functions” that take only what they need, return exactly what they produce, avoid side effects, and are easy to unit test.
- Use “pragmatic functions” as orchestration layers over semantic ones. They change often, deserve concise doc comments about non-obvious behavior, and are best covered by integration tests.
- Model design should make wrong states impossible. Minimize optional fields, avoid loose types, and compose concepts (e.g., UserAndWorkspace) instead of flattening.
- Use brand types (e.g., DocumentId(UUID)) so identical-looking shapes don’t get accidentally swapped.
- Watch the failure modes: semantic functions quietly morphing into pragmatic ones; models bloating with “just one more” optional. Name context-specific functions to signal unstable internals and keep boundaries crisp.

Why it matters: With multiple AI agents touching the same codebase, these guardrails aim to keep correctness, readability, and refactorability from eroding as velocity increases.

### I turned Markdown into a protocol for generative UI

#### [Submission URL](https://fabian-kuebler.com/posts/markdown-agentic-ui/) | 114 points | by [FabianCarbonara](https://news.ycombinator.com/user?id=FabianCarbonara) | [49 comments](https://news.ycombinator.com/item?id=47439300)

Agent builds UIs on the fly: markdown protocol + streaming code + a mount() for React

Hook: Leaning into Eric Schmidt’s “UIs will go away” prediction, this prototype shows an agent that writes and executes code as it chats—generating React interfaces from scratch and wiring data between client, server, and the LLM in real time.

What it is
- A working “agentic UI” demo where the model streams one interleaved feed of text, executable code, and JSON data, and the runtime executes statements as soon as they’re complete.
- The agent mounts live React components inside the chat, updates them via server-side mutations, and streams data directly into them.

The three core ideas
- Markdown as protocol: One stream, three block types:
  - Text: normal markdown for user-facing narration.
  - Code fences: tsx agent.run blocks execute on the server in a persistent context.
  - Data fences: json agent.data => "id" streams JSON directly into mounted UIs.
- Streaming execution: Statements run as they arrive—no waiting for the whole code fence. Implemented with a Bun-based “bun-streaming-exec” wrapper over vm.Script, with top-level await and shared context. “Cursed? Yes. Works? Mostly.”
- mount() primitive: The server executes LLM-written React, serializes it, and the client renders it inline. Data flow remains live and reactive.

The feedback loop
- console.* output and exceptions are fed back to the LLM as a new turn (“runtime transcript”), letting the agent reason about its own execution or wait on user input (e.g., await form.result).

Four data-flow patterns demonstrated
- Client → Server: Forms generated by the agent (zod schemas, {...output.*} bindings) pause execution until submit; results are logged back to the LLM.
- Server → Client: Mutable Data proxies push UI updates immediately (patches over WebSocket).
- LLM → Client: The model streams JSON incrementally (StreamedData + jsonriver) to hydrate lists/loading states as tokens arrive.
- Server ↔ LLM: Execution logs/errors become LLM input; the LLM responds with new code or narration.

Why it matters
- Moves beyond tool-calling to full “interface calling”: the model composes UI, kicks off work early, and updates views as data arrives.
- Piggybacks on what LLMs already know (markdown, React/JSX), avoiding bespoke DSLs.
- Suggests a path to ephemeral, task-specific UIs instead of static, prebuilt screens.

Notable details and trade-offs
- Uses React on the client; Bun and vm.Script for server-side streaming eval; jsonriver for incremental JSON parsing; zod for form schemas.
- Security/sandboxing, determinism, and state/version control are big open questions when executing LLM-written code.
- Developer ergonomics (debuggability, race conditions, latency, cost) will make or break this pattern.

Bottom line
- A compelling, scrappy blueprint for agent-driven interfaces: one stream, immediate execution, and a minimal mount() primitive to turn model output into live UI. The repo and deep-dive on the streaming executor are linked in the post.

#### [Submission URL](https://www.anthropic.com/features/81k-interviews) | 197 points | by [dsr12](https://news.ycombinator.com/user?id=dsr12) | [185 comments](https://news.ycombinator.com/item?id=47435156)

Hacker News Top Story: What 81,000 people want from AI

Anthropic ran a massive, multilingual qualitative study by turning Claude into an “AI interviewer” and asking users worldwide how they use AI, what they hope it enables, and what they fear. In one week, 80,508 Claude.ai users across 159 countries and 70 languages completed open‑ended interviews—likely the largest qualitative study of its kind.

Key findings
- Hope and alarm coexist in individuals: People report real benefits alongside worries about skills, jobs, and societal impacts.
- What people most want from AI (primary desire, per respondent):
  - Professional excellence (18.8%): Offload routine work, focus on higher‑value tasks, better outcomes.
  - Personal transformation (13.7%): Coaching, self‑understanding, emotional support, health.
  - Life management (13.5%): Cognitive scaffolding—schedules, organization, reducing mental load.
  - Time freedom (11.1%): Reclaim time for family, rest, hobbies.
- Lived impacts cut both ways:
  - Help: Faster documentation, better decision support, even aiding misdiagnosis discovery.
  - Harm: Job displacement; fears of over‑reliance and eroding critical thinking.
- Concerns are multi-faceted: Job loss, safety, privacy, misinformation, and unease about systems “smarter than humans.”

How they did it
- Claude conducted semi‑structured interviews and then Claude‑powered classifiers labeled themes (wants, concerns, sentiment, roles).
- Responses were de‑identified; quotes underwent further manual review to remove identifying details.
- An interactive “Quote Wall” lets readers browse voices by region, concern, and vision.
- Appendix details methods and limitations.

Why it matters
- Shifts AI debate from abstractions to concrete, global user aspirations and anxieties.
- Points to strong demand for AI as cognitive copilot: reducing mental load, enhancing mastery, and giving back time—while spotlighting risks that product, policy, and education efforts need to address.

Caveats
- Sample is self‑selected Claude.ai users (not a representative general population).
- Automated classification can introduce labeling bias, especially across languages.
