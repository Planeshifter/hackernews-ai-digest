### LLMs work best when the user defines their acceptance criteria first

#### [Submission URL](https://blog.katanaquant.com/p/your-llm-doesnt-write-correct-code) | 437 points | by [dnw](https://news.ycombinator.com/user?id=dnw) | [390 comments](https://news.ycombinator.com/item?id=47283337)

Headline: LLM-written SQLite “rewrite” looks correct — and is 20,000x slower on a basic query

Summary:
- A practitioner benchmarked system SQLite against a ground-up, LLM-assisted Rust reimplementation (exposed via a C API). On a simple primary-key lookup across 100 rows, SQLite took 0.09 ms; the rewrite took 1,815.43 ms — about 20,171x slower.
- The code compiles, passes tests, reads/writes the SQLite file format, and advertises MVCC, file compatibility, and a drop-in C API. But critical planner and I/O mistakes make it dramatically slower despite looking “right.”

Key findings:
- Even the best-case path (batched transactions) in the rewrite is already 298x slower than SQLite — a bad baseline. Anything higher signals additional bugs.
- SELECT by primary key is ~20,000x slower; UPDATE/DELETE are >2,800x; INSERT without a transaction is 1,857x slower than batched inserts.
- Planner bug: INTEGER PRIMARY KEY isn’t treated as an alias for rowid. The code only recognizes "rowid", "_rowid_", or "oid", so WHERE id = ? triggers a full table scan instead of a B-tree seek. Result: O(n) per lookup and effectively O(n²) behavior across many lookups.
- I/O bug: fsync on every statement (rather than transactional batching) further crushes performance, especially for INSERTs outside a transaction.
- The project is large (≈576k lines, 625 files) with all the “right” components (parser, planner, VDBE, B-tree, pager, WAL) — underscoring how plausible structure can mask fundamental correctness/performance gaps.

Important clarification:
- This is not Turso/libsql. Turso is a fork of SQLite’s C code and, in the same benchmark, performs within ~1.2x of SQLite — consistent with a mature fork, not a reimplementation.

Why it matters:
- LLMs optimize for plausibility, not correctness. You can get compiling, test-passing code that’s disastrously wrong under real scrutiny.
- Define acceptance criteria upfront and verify rigorously (benchmarks, invariants, performance targets). External studies (METR, GitClear) suggest these issues are systemic when LLM output isn’t heavily validated.

Takeaway:
- LLMs are great for speed and scaffolding, but without explicit, measurable requirements — especially for systems code — you may ship something that only looks right.

### Claude Code deletes developers' production setup, including database

#### [Submission URL](https://www.tomshardware.com/tech-industry/artificial-intelligence/claude-code-deletes-developers-production-setup-including-its-database-and-snapshots-2-5-years-of-records-were-nuked-in-an-instant) | 38 points | by [vanburen](https://news.ycombinator.com/user?id=vanburen) | [26 comments](https://news.ycombinator.com/item?id=47287420)

AI agent plus Terraform nukes prod: Developer Alexey Grigorev let Claude Code help migrate his AI Shipping Labs site onto the same AWS stack as DataTalks.Club. Because he kicked off Terraform without the current state file, the bot created duplicate resources. After he later uploaded the state, Claude dutifully reconciled by issuing a terraform destroy—wiping both sites’ infrastructure, a database with 2.5 years of records, and even the snapshots he assumed would be his backups. AWS Business Support restored the data in about a day.

Grigorev’s post-mortem is a useful checklist: stop letting the AI run Terraform; require human review for any destructive plan; move Terraform state to a remote, locked backend (S3 + DynamoDB); enable delete protections and prevent_destroy; separate environments/accounts; and regularly test restores.

The moral isn’t “dumb bot gone wrong” so much as “predictable outcome of missing state + broad permissions + automated execution.” LLMs are very good at following instructions literally—especially the ones that start with destroy.

Based on the discussion, the consensus is that while the AI "pulled the trigger," the incident was ultimately a failure of human oversight and infrastructure hygiene.

**Key themes from the comments:**

*   **The AI Acted Logically:** Several users noted the AI wasn't "wrong"; it strictly followed the logic of the Terraform state file (or lack thereof). As one user put it, "the agent followed the state file logically... it wasn't wrong, it wasn't checked."
*   **Verification Fatigue:** Commenters criticized the author’s post-mortem suggestion to rely on manual CI/CD reviews. They argued that requiring humans to review every step leads to "rubber stamping," where developers approve changes out of habit without reading them. A better approach is to implement systems that specifically flag destructive changes (wipes/deletes) for review while automating the rest.
*   **The Dangers of "Agentic" Ops:** There was significant skepticism about giving unpredictable, probabilistic LLMs direct terminal access to live environments. Users noted using `sudo` or allowing agents to execute CLI commands in production is "extremely obvious" risky behavior.
*   **Accountability vs. Capability:** The discussion touched on the irony of executives claiming AI will replace developers. One user noted the key difference is accountability: when a human wipes a database, they face liability and learn from it; an AI does not.
*   **Infrastructure Fundamentals:** The root cause was identified not as an AI failure, but as "sloppy infra management." Users pointed out that Terraform is notoriously tricky (even for humans) regarding resource replacement rules. More importantly, the deletion of snapshots highlighted the critical need for *offsite* backups—backups that exist outside the AWS account or logic control of the primary stack.
