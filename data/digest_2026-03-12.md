### Document poisoning in RAG systems: How attackers corrupt AI's sources

#### [Submission URL](https://aminrj.com/posts/rag-document-poisoning/) | 143 points | by [aminerj](https://news.ycombinator.com/user?id=aminerj) | [56 comments](https://news.ycombinator.com/item?id=47350407)

A new hands-on lab shows how trivial it is to corrupt a Retrieval-Augmented Generation system by seeding its vector database with a few “authoritative” fakes. On a local MacBook (no GPU, no cloud), the author added three crafted docs to a tiny ChromaDB store and got a Qwen2.5-7B RAG pipeline to assert a company’s Q4 2025 revenue was $8.3M (with layoffs and SEC interest) instead of the real $24.7M with $6.5M profit—consistently across runs.

Key idea
- PoisonedRAG’s two conditions in action: (1) Retrieval—use vocabulary-engineered terms (“Q4 2025,” “Financial Results,” “CORRECTED,” “CFO Office”) so poisoned docs outrank the legit one; (2) Generation—authority framing (“CFO-approved correction,” “regulatory notice,” “board notes”) nudges the LLM to treat the fake as current truth.
- Even without gradient-optimized payloads (as in the cited USENIX Security 2025 paper), three coordinated docs can dominate top-k and push the real financials out of context.

Why it matters
- RAGs inherit the integrity (and write permissions) of their knowledge bases. If an internal wiki or vector store is weakly governed, “soft” language cues plus retrieval bias can silently flip answers. Prompt hardening helps but doesn’t fix poisoned recall.

Defensive takeaways
- Treat KB integrity as a security boundary: strict write access, versioning, signing/attestation, and audit trails.
- Harden retrieval: diversify top-k, re-rank with cross-encoders and trust signals, penalize sudden “corrections” lacking provenance, deduplicate/cluster near-duplicates.
- Add contradiction and outlier checks across sources; pin high-trust canonical docs in context for critical entities/metrics.
- Surface provenance to users; run adversarial evals and canary docs to detect poisoning early.

Repo: aminrj-labs/mcp-attack-labs (lab 04) demonstrates the mechanics for defenders.

**Mitigation and Defense Strategies**
Much of the conversation focuses on how to defend against these attacks, with the submission author (**mnrj**) actively debating technical nuances with other users.
*   **Backtesting vs. Runtime Monitoring:** User **ndsrnm** argues that RAG systems should be treated with traditional Model Risk Management (MRM), including strict backtesting and "vetting" of the document store. **mnrj** pushes back, noting that unlike traditional ML, RAG data is mutable at runtime; a static backtest doesn't protect against a bad actor adding a file five minutes later.
*   **Provenance and Metadata:** **ctsftwr** suggests that strict linking of document chunks to "owners" (e.g., Active Directory identity) allows for filtering non-authoritative sources. **hrmtst93837** counters that this isn't a silver bullet; if an attacker compromises a legitimate user (phishing/social engineering), they can upload "junk knowledge" with a valid signature.
*   **Anomaly Detection:** **nzch** discusses the practicality of embedding anomaly detection (spotting chunks that don't fit), though they worry about false positives in naturally overlapping documents (like transcripts). They suggest a "self-healing" database approach rather than blocking inputs. **mnrj** notes that clustering analysis can help identify "adversarial" documents that arrive in tight batches without needing to read the content.

**The Nature of LLM Limitations**
*   **Lack of "Truth":** **kpw94**, **tlnd**, and others point out a fundamental flaw: LLMs have no concept of objective truth (Good vs. Bad), only text prediction. If the input data ("Garbage In") contains lies—whether about a flat earth or financial fraud—the model will output them as fact ("Garbage Out").
*   **Context Decay:** **ndsrnm** and **mnrj** discuss "context decay." As context windows fill up, models (especially smaller ones like the Qwen model used in the lab) lose the ability to track contradictions, making them easier to poison. **mnrj** hypotheses that "reasoning" models (which "think" before answering) might be better at spotting these contradictions.

**Comparisons and Use Cases**

*   **Natural Defenses:** **XR843** shares a fascinating use case involving 11 million characters of classical Buddhist texts. They find that because the corpus contains multiple independent editions (Chinese, Sanskrit, Pali), the system has a built-in cross-referencing capabilities that naturally resist poisoning using just one source. **mnrj** notes this is essentially a natural form of redundancy/anomaly detection.

### Are LLM merge rates not getting better?

METR measured two success criteria—“passes all tests” vs. “maintainer would approve”—and found performance drops sharply under the stricter, maintainer-approval bar. The post fits simple trend models to METR’s reported merge rates using leave-one-out cross-validation and Brier scores: a constant rate fits best (0.0100), a step function (late‑2024 jump then flat) is next (0.0117), and a gentle upward slope—METR’s visual takeaway—performs worst (0.0129). The author’s read: after a possible step up in late 2024, merge rates have been flat since early 2025, implying LLM programming ability has plateaued despite ongoing hype. They note rumors of fresh gains from newer Anthropic/Google models, but say there’s no equally careful, public measurement beyond METR’s Sonnet 4.5 data to confirm it.

Based on the discussion, commentors largely debated the nature of software quality, the validity of "merge rate" as a metric, and the economic implications of AI in big tech.

**Quality, Maintainability, and the "S-Curve"**
Many users agreed with the submission's sentiment that LLM coding capabilities may have plateaued or hit an "S-curve." Several practitioners noted that while models can generate code that passes reviews or tests, the *quality* is often terrible.
*   Users described LLM-generated code as having "quadruple-nested control flow," absurd architectures, and "mega functions" essentially copy-pasted with minor changes.
*   One commenter noted that fixing a bug in LLM code is difficult because it duplicates logic across the codebase; fixing it in one place doesn’t fix it in ten others.
*   A distinction was drawn between "passing a test" and "maintainable code." Users argued that even if merge rates look flat, the *context* management (e.g., 200k+ token windows) has improved, allowing for more complex, albeit messy, interactions.

**The "Code Review" Debate**
A sub-thread debated whether human review of LLM code is necessary.
*   Some compared LLM code to compilation targets (like Assembly code) or "results" from a calculator—arguing that as long as the output (the running software) works, the internal code structure shouldn't matter.
*   Others strongly disagreed, arguing that unlike "regenerating from scratch," software requires maintenance. Therefore, intermediate code must be readable, architecturally sound, and unit-tested to avoid subtle logic bugs that models frequently introduce.

**Hidden Progress and "All-or-Nothing" Metrics**
Reviewers suggested the "merge rate" metric might hide genuine progress in workflow efficiency.
*   One user argued that the real improvement is in the reduction of "back-and-forth" clarifications. Models now understand intent and scoping better, even if they don't produce a perfect merge-ready PR on the first shot.
*   Others pointed to the "emergent abilities" theory: complex tasks are often "all-or-nothing" (like a self-driving car). A model that yields a 99% correct code block often fails completely due to one error, making progress look flat on a graph until it suddenly jumps (a step function).

**Bureaucracy vs. Automation**
The conversation shifted to the labor impact. Participants debated whether AI would automate white-collar jobs or if "BS jobs" would persist regardless.
*   Using Meta's 80k headcount as an example, users debated whether massive tech workforces are actually doing productive work or if they merely exist due to organizational complexity and "cheap money."
*   Some argued that while AI might automate coding, the "coordination costs" (legal compliance, ad optimization, internal reporting) keep headcounts high, meaning AI might not lead to mass unemployment even if it solves coding.

- Executable specs provide a tight, deterministic contract you can test code against.
- Workflow: start from an existing Quint spec and an English description; have AI draft spec changes; iterate with parse/typecheck for structural soundness; then humans focus on behavioral validation.
