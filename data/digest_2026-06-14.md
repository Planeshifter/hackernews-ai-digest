### Don't trust large context windows

#### [Submission URL](https://garrit.xyz/posts/2026-05-06-dont-trust-large-context-windows) | 267 points | by [computersuck](https://news.ycombinator.com/user?id=computersuck) | [188 comments](https://news.ycombinator.com/item?id=48524620)

Don’t trust large context windows: The author argues that LLMs have a “smart zone” and a “dumb zone” inside their context window, with useful attention tapering off around ~100k tokens. Beyond that, models start forgetting and mis-weighting earlier instructions—even if the vendor advertises 200k–2M tokens.

Key points:
- Agents hit the dumb zone fast: a few file reads, verbose logs, and test runs can blow past 100k before lunch.
- Evidence backs it up: studies like RULER and Chroma’s “context rot” show performance degrades as the window fills; the effective working set is a fraction of the headline number.
- Auto-compaction helps but isn’t a cure: tools (e.g., Claude Code) summarize and reset, but they trigger after degradation and the summaries are produced under degraded attention.
- Practical fix: restart sessions and hand the model a human-written spec or artifact (PRD, plan, skill definition). “Breadcrumb” the work so each new session starts in the smart zone.
- Workflow pattern: structure agents around small, named artifacts and explicit handoffs (e.g., obra/superpowers, mattpocock/skills) to keep live context tight.
- Mental model: treat context like a budget—the first chunk does the real work; push everything else into durable, external artifacts the next session can read.

Because LLMs are opaque black boxes, developers resort to superstitious "cargo culting"—doing things just because they seemed to work once.

*   **Toolchain Compatibility:** The project's creator noted that YOLOv8 acts as a practical, "known-good" baseline. It possesses well-supported export paths for Rockchip’s RKNN toolchain (specifically for ONNX to INT8 conversion). Newer YOLO versions currently require too much manual export fixing and troubleshooting to be viable out of the box. 
*   **Licensing Issues:** Other commenters pointed out that the newer YOLO models suffer from murky, restrictive licensing (moving away from true open-source), which discourages enthusiast and open-source development. Users suggested that developers looking for truly open, modern alternatives should investigate **RT-DETR** instead.

**NPU Accessibility**
Overall, the project sparked a strong appreciation for accessible hardware. Commenters expressed excitement about the ability to program and hack on these low-cost NPUs, noting how refreshing it is compared to the heavily locked-down, proprietary ecosystems of major consumer tech giants where NPU access is often restricted.
