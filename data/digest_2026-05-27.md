The payoff, per Boris Cherny: a 2–3x quality jump once you give Claude a way to verify its own work.

Why it matters
- Moves you from ad-hoc prompting to a compounding, team-shared setup.
- Shifts Claude from pair-programming to true delegation with plans, reviews, and self-updating rules.
- Explains the .claude directory as a layered, path-aware system you can actually engineer.

Key tactics from the guide
- Verification first: Don’t be the only feedback loop. Build in checks so Claude can test and iterate on its own.
- Explore → plan → code: Use Plan mode (Shift+Tab twice) to read and map the system before implementing. For multi-file changes, get a plan, then execute.
- Dual-claude planning: Write a plan in one session; spin up a clean session to review it “as a staff engineer” to catch gaps without context bias.
- Reference, don’t describe: Point Claude at exact files (@src/auth/login.py) and pipe logs/errors (cat error.log | claude) instead of paraphrasing.
- Delegate, don’t co-drive: Brief crisply, then let it run. Treat it like an engineer you manage, not a line-by-line pair.
- Edit the plan mid-flight: Ctrl+G opens the plan in your editor so you can shape it before code is generated.
- Institutionalize learning: When Claude errs, end with “Update CLAUDE.md so you do not repeat this.” It’s “eerily good” at distilling its own rules.

The .claude directory, demystified
- Two scopes:
  - Project: .claude/ in the repo, checked in for team sharing.
  - Global: ~/.claude/ for machine-wide defaults.
- Important files:
  - CLAUDE.md (project/global, committed): session-start instructions that cascade (monorepo-friendly).
  - CLAUDE.local.md (project, gitignored): your private notes/preferences for that project.
  - settings.json + settings.local.json: shared defaults vs personal overrides (permissions, hooks, env vars, model prefs).
  - .mcp.json: team-shared MCP servers.
  - rules/*.md: path-gated, topic-scoped guidance so you don’t bloat CLAUDE.md.
  - commands/*.md vs skills/<name>/SKILL.md: both create slash commands, but skills are richer (supporting files, tool/agent controls). Prefer skills for new work.
  - agents/*.md: subagent definitions for specialized workflows.
- Handy op: claude project purge ~/path/to/repo --dry-run shows exactly what Claude is caching before you hand off a machine.

How to write CLAUDE.md (Boris’s approach)
- Keep it short: If removing a line wouldn’t cause mistakes, cut it.
- Let Claude refine it: After failures, have Claude update the file with new rules gleaned from its own missteps.

Scope of the guide
- Beyond setup, the ToC covers: real-world skills, building custom subagents (e.g., /pr-review), plugins/marketplace, underused commands like /goal, MCP-powered workflows (e.g., with Obsidian), and daily workflow optimization, plus examples worth studying.

Who should read this
- Engineers and teams already using Claude Code who want reproducible, higher-signal sessions and a shared, evolving project brain—especially in monorepos or complex services.

Takeaway
Stop chatting; start engineering your agent. Put plans, verification, path-gated rules, and reusable skills at the center—and make CLAUDE.md a living, auto-improving spec.
