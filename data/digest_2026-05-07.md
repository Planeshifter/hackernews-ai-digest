### Agents need control flow, not more prompts

#### [Submission URL](https://bsuh.bearblog.dev/agents-need-control-flow/) | 556 points | by [bsuh](https://news.ycombinator.com/user?id=bsuh) | [270 comments](https://news.ycombinator.com/item?id=48051562)

Agents need control flow, not more prompts (May 7, 2026)

The gist:
- Prompt chains hit a hard ceiling for complex, reliable work. If you’ve ever written “MANDATORY” or “DO NOT SKIP,” you’ve felt it.
- Software scales via deterministic control flow and composability; prompts are non-deterministic, weakly specified, and hard to verify.
- Treat the LLM as a component, not the system: move logic out of prose and into code that enforces explicit state transitions and validation checkpoints.

Why it matters:
- As tasks get complex, “vibes” don’t compose. Without predictable control flow, local reasoning breaks and reliability collapses.
- Deterministic orchestration is necessary but insufficient—silent failures demand aggressive error detection and programmatic verification.

Key points:
- Analogy: a programming language where statements are suggestions and functions hallucinate “Success” breaks reasoning entirely.
- Reliability requires deterministic scaffolds: explicit states, transitions, and validations at runtime.
- In the absence of verification, you’re left with:
  - Babysitter: keep a human in the loop to catch errors.
  - Auditor: verify everything after the fact.
  - Prayer: accept outputs on faith.

Bottom line:
- Stop piling on prompts; encode control flow and checks in software. Build agents like software systems—code all the way down—with LLMs as fallible components guarded by state machines and validators.

# Hacker News Daily Digest: The Shift from "Vibes" to Deterministic AI

**Today’s Top Discussion:** *Agents need control flow, not more prompts.* 
**The Consensus:** Hacker News commenters overwhelmingly agree with the submission. The era of trying to coax complex, multi-step reliability out of mega-prompts is ending. Developers are finding that putting LLMs in charge of high-level logic, loops, and orchestration is a recipe for silent failures, blown budgets, and infinite debugging loops.

Here is a breakdown of the key themes, insights, and debates from the comment section:

### 1. The Breakdown of "Prompt-Only" Workflows
Commenters shared painful anecdotes of trying to force LLMs to manage states and loops. One user noted an attempt to have an AI agent process 200 markdown files containing browser session requirements. By relying on the AI to manage the control flow, the system started breaking after just 30 files—skipping some files entirely and re-testing others unnecessarily. 
* **The fix?** Building a "super basic deterministic harness" in standard code that loops through the files and explicitly triggers the LLM for individual checks, saving state along the way.
* **The golden rule:** As one user aptly put it, *"Don't tell the LLM to loop over files. Tell the LLM to write a script with a loop, and run the script."*

### 2. Standard Code Unlocks Cheaper, Smaller Models
A major revelation in the thread is the cost-to-performance ratio. When you force a state-of-the-art (SOTA) model (like Claude 3 Opus or GPT-4) to act as both the orchestrator *and* the worker, you pay premium token prices for fragile results.
However, when you encode the orchestration (if-statements, loops, logic) in Python or another deterministic language, the actual AI tasks become small, isolated, and highly explicit. Commenters noted that when tasks are broken down this way, cheaper, smaller models (like Qwen or LLaMA) deliver results comparable to the giant, expensive models.

### 3. Deep Skepticism Toward "Managed Agent Platforms"
There is a growing frustration with black-box agent platforms (users name-dropped tools like Cursor Cloud Agents and Anthropic's managed orchestration). Developers complain that these platforms abstract away the control flow, trying to manage sub-agents via hidden prompt chains rather than giving developers explicit, deterministic hooks. 
* Many users advocate replacing these "black box" platforms with simple, custom-written `skill.py` tasks glued together by standard CLI orchestration, or utilizing frameworks like Google ADK (v2) which guarantees strict graph routing and control flow.

### 4. Code as the Ultimate "Validator"
Users stressed that building reliability means relying on compilation, unit tests, and structural checks (like Playwright for browser tasks) to validate the AI's output at runtime. 
* One user highlighted achieving an 80% success rate on code generation by hooking the AI explicitly to deterministic tools: `apply_patch`, `check_compilation`, and `run_unit_tests`. If the code compiles and passes tests, the state machine moves forward. 

### 5. The "Magic" Trap 
The thread delved into why the industry got stuck on prompts in the first place: non-technical users view AI as "magic." Because non-developers cannot easily write the Python scaffolding to loop, parse, and validate, they demand the LLM do it all in natural language. Conversely, software engineers recognize that combining a fallback-prone LLM with the rigid, infallible execution of a traditional programming language yields the best of both worlds.

### The Bottom Line
Stop treating LLMs as the operating system. Treat them as a simple, fallible function call within your code. Software engineering fundamentals—state machines, explicit routing, deterministic testing, and modularity—are still the only way to build reliable, scalable systems.
