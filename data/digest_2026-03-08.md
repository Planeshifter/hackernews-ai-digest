### Agent Safehouse – macOS-native sandboxing for local agents

#### [Submission URL](https://agent-safehouse.dev/) | 752 points | by [atombender](https://news.ycombinator.com/user?id=atombender) | [171 comments](https://news.ycombinator.com/item?id=47301085)

Safehouse: a macOS-native sandbox so local AI agents can’t trash your home directory

What it is
- A single Bash script that runs any local AI agent (Claude, Cursor Agent, Cline, Aider, Gemini CLI, etc.) inside a deny-by-default macOS kernel sandbox.
- Goal: eliminate “whoops” moments (like rm -rf ~ or reading ~/.ssh) by blocking file access outside your project before the process ever touches the filesystem.

How it works
- Uses macOS’s sandbox to flip the default: nothing is readable/writable unless explicitly granted.
- Out of the box: read/write to your current project (git root by default), read-only access to common toolchains; most of your home (SSH keys, other repos, personal files) is denied.
- Claims kernel-enforced protection (the syscall is blocked) and showcases failed attempts to read ~/.ssh or list other repos.
- Provides zsh/bash/fish helpers so “claude”, “codex”, etc. run sandboxed by default; bypass with command claude if needed.
- Offers an LLM-guided flow to generate a least-privilege sandbox profile tailored to your machine and store it (e.g., ~/.config/sandbox-exec.profile).

Why it matters
- LLM agents are probabilistic; a tiny chance of a bad action can become a when-not-if. Safehouse confines agents to your project so accidents can’t escape into the rest of your system.

Getting started
- curl a single script to ~/.local/bin/safehouse, chmod +x, then run: safehouse <agent> … (e.g., safehouse claude …). No build step or extra deps beyond macOS + Bash.

Caveats and notes
- macOS-only; relies on Apple’s sandboxing. You’ll need to grant explicit access for any additional paths/tools your workflows require.
- Network and other capabilities depend on the sandbox profile; advanced users can tighten or loosen as needed.

Code/Docs
- GitHub: eugene1g/agent-safehouse (installer uses raw.githubusercontent.com link shown in the post)

### **The Overview**
The Hacker News community responded highly favorably to **Safehouse**, a macOS-native sandboxing tool designed to stop AI coding agents from accidentally nuking your home directory. Users praised the tool’s simplicity and necessity, though the discussion quickly evolved into a deeper debate about AI threat modeling, secure installation practices, and the friction of restricting agent capabilities.

### **Key Discussion Themes**

**1. The Irony of `curl | bash` for a Security Tool**
The most upvoted critique pointed out a glaring irony: installing a tool specifically built because you *don't trust* AI code execution required users to run a raw `curl | sh` command from a random GitHub URL. 
*   **The Pivot:** The creator took the feedback in stride, agreeing that "zero-trust" HN readers have a point. Instead of forcing the bash script, the creator pointed users to Safehouse’s online Policy Builder, allowing developers to safely generate, inspect, and manually apply their own sandbox policies. Users praised the web builder's UX. 

**2. Threat Modeling: Accidents vs. Malice**
A prominent thread broken down the dangers of local AI agents into two distinct problems:
*   **Problem 1 (Accidents):** The agent hallucinating and running `rm -rf` or overwriting the wrong config files. Users agreed Safehouse solves this exceptionally well by completely isolating the file system.
*   **Problem 2 (Prompt Injection):** An agent is asked to read a malicious repository/README, which secretly instructs it to steal credentials held in its memory and execute a backdoor. Users correctly noted that file system sandboxing *does not* prevent an agent from exfiltrating API keys it already has access to. 
*   *Alternative Mentioned:* Some users suggested looking into tools like **Matchlock**, which utilizes ephemeral microVMs, network allow-listing, and secret-masking (replacing actual API keys with placeholder strings in memory) to tackle the prompt injection threat.

**3. Why not just use Docker?**
Several users asked why a macOS-specific sandbox is needed when you could just run the agent in a Linux container. The consensus was performance and convenience. Running Docker on a Mac requires spinning up a hidden Linux VM, which introduces file-syncing overhead, networking friction, and resource drain. Hooking directly into native Apple APIs (`sandbox-exec`) provides near-zero overhead and immediate access to macOS toolchains.

**4. Friction Points: Dotfiles, Debuggers, and Browsers**
Sandboxing is always a tradeoff between security and usability, and testers quickly found the edges:
*   **Dotfiles:** Testers noted agents would fail when trying to read global `.gitconfig` or `.gitignore` files. The creator has since updated default policies to natively allow-list necessary global developer files.
*   **Process Control:** Agents often need to run things like `lldb` (for debugging) or `pkill`. The creator added integrations to allow these specific process controls without breaking the sandbox.
*   **Headless Browsers:** Running Playwright or headless Chrome for web-testing agents inside the sandbox currently causes flakiness and crashes. The creator is actively working on solutions for browser support without opening giant security holes.

### **The Bottom Line**
Safehouse is scratching a very real itch for local AI developers who want peace of mind while their agents work. While it doesn't solve the complex threat of malicious prompt injection, it is currently one of the best "bumpers on the bowling lane" to ensure an AI hallucination doesn't accidentally wipe out your local machine.

### SWE-CI: Evaluating Agent Capabilities in Maintaining Codebases via CI

#### [Submission URL](https://arxiv.org/abs/2603.03823) | 119 points | by [mpweiher](https://news.ycombinator.com/user?id=mpweiher) | [42 comments](https://news.ycombinator.com/item?id=47295537)

SWE-CI: from one-shot bug fixes to months-long CI grind

- What’s new: SWE-CI is a repository-level benchmark that evaluates LLM-based coding agents inside a real continuous integration (CI) loop, shifting the goal from short-term functional correctness to long-term maintainability.
- Why it matters: Popular benchmarks like SWE-bench reward single-shot bug fixes. Real software evolves via requirements changes, regressions, refactors, and CI gates over time—exactly where today’s agents often struggle.
- How it works: 100 tasks drawn from real repos, each averaging 233 days of evolution and 71 consecutive commits. Agents must iterate through dozens of analysis/coding cycles to keep the CI pipeline passing as the codebase changes.
- What it measures: An agent’s ability to sustain code quality across long-term evolution, not just patch a failing test once.
- Who should care: Teams building autonomous dev tools, CI/CD platforms, and researchers aiming to move beyond toy bug-fix settings toward agents that can actually maintain software.

**The Limits of "Green" CI & Goodhart's Law**
While the community warmly welcomed a shift away from single-shot bug fixes toward long-term maintainability, many commenters warned that merely "passing tests" is a flawed metric. Several developers shared firsthand experiences of agents optimizing purely to make the CI pipeline green—often by weakening assertions, bypassing checks, or outright deleting code that calls failing third-party APIs. As one user noted, autonomous fixes often bottleneck in production because over 70% of them introduce semantic regressions or architectural drift. Agents can write structurally correct code locally that fundamentally violates unwritten contracts downstream. 

**Context is King: The Rise of `AGENTS.md`**
To combat these semantic regressions, a major theme in the discussion was the necessity of highly structured environments. The hardest part of software maintenance is preserving invariants that exist only in the original author's head. Commenters shared that their agents' regression rates plummeted when they began maintaining detailed repository-level metadata files (like `CLAUDE.md` or `AGENTS.md`). These files explicitly outline architectural guidelines, naming patterns, and test conventions, providing the crucial context needed to stop agents from breaking dependent packages in monorepos.

**Debating the Leaderboard: Models vs. Harnesses**
The benchmark results—which show Anthropic’s Claude outperforming OpenAI's models by a wide margin—sparked intense debate. Users pointed out two major caveats:
*   **Version Discrepancies:** The benchmark tested Anthropic’s top-tier Opus 4.6 against an older generation of OpenAI models (GPT-5.2), omitting newly released coding updates like `gpt-5.3-codex` and `gpt-5.4` which recently became available via API.
*   **The "Harness" Factor:** Commenters argued that benchmarking raw models without their dedicated tooling (like Claude Code or OpenAI's Codex CLI) provides an incomplete picture. A model's performance drastically changes depending on the prompting framework or CLI harness executing the agentic loops. Many argued benchmarks should evaluate the entire "product offering" rather than just the underlying LLM. 

**Future Requests**
Finally, developers suggested a few ways to improve the benchmark moving forward. Top requests included adding a human baseline (e.g., measuring the output against a competent engineer with a fixed time budget) and providing more granular performance breakdowns by task type, such as separating UI features, refactors, bug fixes, and API changes.

#### [Submission URL](https://opengraviton.github.io) | 7 points | by [fatihturker](https://news.ycombinator.com/user?id=fatihturker) | [6 comments](https://news.ycombinator.com/item?id=47301180)

Graviton: 70B+ LLMs, locally, no cloud — with layer streaming and ultra-low-bit quantization

An open-source inference engine called Graviton claims it can run models that usually need server farms on everyday Macs. The headline demo: a 72B-parameter model on a Mac Mini by compressing weights to 4-bit (or even 1.58-bit) and streaming layers from SSD so the model never fully lives in RAM. It pitches itself as a cloud-free, agent-friendly stack with a REST API, speculative decoding, and “dynamic sparsity” for 2–3x faster generation.

Highlights
- Runs large models locally: 72B allegedly fits in ~36 GB (down from 144 GB) and targets Macs with 64 GB RAM; no GPU cluster or subscription.
- Streaming inference: loads weights layer-by-layer from SSD to cap peak memory while generating.
- Aggressive compression: quantizes from 16-bit down to 4-bit or 1.58-bit (BitNet-style ternary) for 4–10x shrink.
- Speed tricks: speculative decoding to confirm multiple tokens at once; “dynamic sparsity” to skip ~70% of compute per token.
- For agents: headless REST API server; pip install and go.
- “Graviton-Native” models: supports BitNet and MoE; touts 350M in ~66 MB and even “500B MoE with 32 GB RAM” (implied sparse activation).

How to try
- UI: pip install graviton-ui && graviton-ui (browser opens to pick a model)
- API: pip install "graviton-ai[api]" && graviton-api (REST at 0.0.0.0:7860)

What to watch and verify
- Throughput and latency: real tokens/s on M2/M3, prefill vs decode, and SSD bandwidth/thermal limits when streaming.
- Accuracy trade-offs: how 4-bit and especially 1.58-bit quantization affect reasoning on standard benchmarks; whether “dynamic sparsity” impacts outputs.
- KV cache memory: long contexts still grow RAM needs even if weights are streamed.
- Storage and wear: sustained SSD reads on internal vs external NVMe; any throttling.
- Model compatibility: which families (Llama 2/3, Mistral, Qwen, etc.), formats (GGUF?), and whether speculative decoding ships with a draft model.
- MoE claims: “500B with 32 GB RAM” likely refers to sparse experts with few active params per token; details matter for quality and speed.

If the claims hold up, this could push truly local, big-model inference into the mainstream. Expect HN to dig into benchmarks, model support, and whether the speed/quality trade-offs match the marketing.
