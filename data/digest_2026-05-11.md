### I let AI build a tool to help me figure out what was waking me up at night

TL;DR: In a noisy city, the author built a local-only system that records short audio clips when loud sounds occur and lines them up with wearable sleep data and home sensors. AI didn’t “solve sleep,” but it made the whole build feasible in a weekend—lowering the cost of crafting a bespoke tool for a real-life problem.

What they built:
- Hardware: Two cheap USB mics (one indoors, one facing the street) on a Raspberry Pi.
- Privacy gating: The Pi only “listens” when Home Assistant detects the author is home, in bed, and within usual sleep hours; otherwise mics are fully disabled.
- Detection: Threshold-based audio trigger saves clips with a few seconds of pre/post context.
- Data fusion: A web app shows DAW-style tracks—sleep stages, heart rate/HRV, sensor events, and noise events—with easy zoom and instant playback.
- Anchors: Garmin watch data marks wake moments and stage transitions, serving as review waypoints (good at “you woke up here,” not clinical staging).
- UX: Progressive web app on the home network with morning push notifying that last night’s timeline is ready. Nothing leaves the LAN.

How AI helped:
- Accelerated coding and setup: An agent wrote most of the app and Pi code; the author focused on testing and feedback rather than reading code.
- Device-level iteration: The agent got SSH access to the fresh Pi, ran on-device experiments (prompting the author to make noises), and even produced spectrograms.
- Not doing the “smart” part: No AI sound classification yet; the human still listens to flagged clips. The win was speed and lowered effort, not magical detection.

Results so far:
- Clear culprits surfaced quickly—especially sharp transients like door slams and dishes—that correlate with wake markers and light-sleep arousals.
- The tool turns fuzzy 3 a.m. mysteries into actionable events without sending any audio to the cloud.

Why it matters:
- A practical pattern for HN readers: fuse cheap sensors + Home Assistant + wearables into a local, privacy-preserving “flight recorder” for everyday annoyances.
- The interesting shift: AI agents meaningfully reduce the time/complexity to build personal tools, empowering individuals to investigate and fix their own quality-of-life issues.

### Natural-language messages between LLM agents are an architectural anti-pattern

#### [Submission URL](https://novaberg.de/papers/clipboard-pattern.html) | 21 points | by [ClausVomBerg](https://news.ycombinator.com/user?id=ClausVomBerg) | [5 comments](https://news.ycombinator.com/item?id=48099171)

TL;DR
Claus Schlehhuber argues that the default “agents talk to each other in prose” pattern is an architectural anti‑pattern. Every hop re-encodes meaning, burns tokens, and defeats unit testing. His fix: a single, shared, typed state object — a “clipboard” — that flows through a LangGraph. Each specialist node reads the fields it needs, writes structured results back, and hands the clipboard on. Transport data completely; let the consumer format.

What’s new
- Names and sharpens a practice many teams have converged on: replace natural‑language message passing with a strongly typed, shared state (TypedDict schema) inside a graph.
- Introduces a clean, scalable taxonomy — Roles, Departments, Graphs — for composing many specialists without devolving into prompt spaghetti.
- Open implementation notes in LangGraph; Apache 2.0.

Why it matters
- Semantic drift: Each agent rewording another agent’s text accumulates error; intent turns into rumor after a few hops.
- Token cost: You repeatedly pay to read and rewrite the same facts in paragraphs instead of a few schema fields.
- Testability: Strings-in/strings-out agents resist unit tests. A typed clipboard enables fixtures and precise assertions on fields and provenance.

How it works
- One shared “case file” (clipboard) moves from node to node. Nodes:
  - Read only the typed fields they care about.
  - Write results to specific fields (with types, structure, and provenance).
  - Never summarize each other’s outputs in prose.
- At the end, you format from the clipboard once for the user or downstream system.

Context and critique
- Calls out CrewAI, AutoGen, and common LangChain “agent-supervisor” templates for defaulting to NL message passing; says the fix isn’t better prompts but a different architecture.
- This is essentially a modernized blackboard architecture: not novel, but well articulated and named for today’s LLM tooling.

Trade-offs and gotchas
- Upfront schema design, versioning, and migrations add discipline (and friction).
- Concurrency/merging become real design questions if multiple branches update the clipboard.
- You still need robust parsing/validation to coerce model output into the schema.
- Best fit for pipeline/graph workflows; free-form “chatty” collaboration is intentionally de-emphasized.

Sound bite
“Multi-agent systems shouldn’t gossip. They should share a case file.”

For practitioners
- If your agents are summarizing each other’s summaries, expect drift, higher spend, and opaque failures.
- Move to a typed shared state, add validators, track provenance per field, and make agents unit-testable with fixtures.
- Keep natural language at the edges (I/O); keep structure inside the system.

**The Core Debate: What is an LLM’s "Native" Format?**
The primary debate in the thread centers on whether structuring data (as suggested by the article) fights against the fundamental nature of Large Language Models.

In a separate thread, a commenter strongly reinforced the article's point regarding software testing. From a software engineering perspective, plain strings simply cannot serve as valid test vectors for system validation, making the case for a strongly typed "clipboard" state much more appealing. 

**Summary of Sentiment:** 
While there is some philosophical debate over whether rigid data structures ignore the text-native strengths of LLMs, the more technical consensus leans toward the article's premise: passing raw text between non-deterministic models causes data loss and ruins testability.
