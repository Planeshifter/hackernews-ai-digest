### Real-time LLM Inference on Standard GPUs: 3k tokens/s per request

#### [Submission URL](https://blog.kog.ai/real-time-llm-inference-on-standard-gpus-3-000-tokens-s-per-request/) | 208 points | by [NicoConstant](https://news.ycombinator.com/user?id=NicoConstant) | [94 comments](https://news.ycombinator.com/item?id=48321076)

Title: Kog: Hitting “thousands of tokens/sec” on standard GPUs by optimizing for single-request decode speed

What’s new
- Kog claims you can reach dedicated-inference–card speeds on off-the-shelf datacenter GPUs by co-designing model architecture, runtime, and low-level GPU kernels for single-request decoding.
- Public tech preview: a 2B coding model you can try at playground.kog.ai (batch size 1, latency-first).

Why it matters
- Autonomous agents are bottlenecked by serial loops (inspect → plan → edit → test → revise), so per-request decode speed—not aggregate throughput—sets UX and product ceilings.
- Example: 50k tokens at 100 tok/s ≈ 8 minutes; at 3,000 tok/s ≈ 20 seconds—enabling very different agent behaviors and tooling.
- If standard GPUs can deliver this without proprietary silicon, buyers avoid vendor lock-in.

Core argument
- For batch size 1, LLM decoding is bandwidth-bound, not FLOPS-bound.
  - Low arithmetic intensity: FP16 weights are ~1 FLOP/byte; FP8 ~2; FP4 ~4, vs GPUs offering hundreds of peak FLOPs/byte of HBM bandwidth.
  - Therefore, Memory Bandwidth Utilization (MBU) is the key metric; Model FLOP Utilization (MFU) matters mainly when batching, which hurts latency.

Key numbers (theoretical ceilings, ignoring KV/cache reload penalties)
- 8× NVIDIA H200 node: ~30.7 TB/s effective HBM bandwidth.
- 8× AMD MI300X node: ~33.6 TB/s effective.
- 2B FP16 dense (~4 GB active weights): 
  - H200: ~7,700 tok/s 
  - MI300X: ~8,400 tok/s
- MoE examples at batch size 1:
  - 4B active FP8 ≈ same regime as above.
  - 32B active FP4 ≈ ~2,000 tok/s.
- Next-gen GPUs (Rubin, MI450) are expected to 4× bandwidth, enabling equivalent speeds on ~4× larger models or fewer GPUs.

How they get there
- Treat an entire 8-GPU node as a single latency-optimized pipeline to maximize effective HBM bandwidth for one request.
- Co-design across stack layers: architecture (including MoE), runtime/engine, and hand-tuned GPU kernels to minimize reloads, improve cache use, and reduce KV traffic (the β factor).
- Run batch size 1 in the preview to prove ceiling; batching will come later for throughput targets.

Takeaway
- If you optimize for MBU and co-design the full stack, single-request decoding in the thousands of tokens per second on standard datacenter GPUs looks achievable—potentially reshaping what agentic software can do in real time. Test their 2B coding model at playground.kog.ai.

### CAPTCHAs can still detect AI agents

#### [Submission URL](https://research.roundtable.ai/captchas-detect-ai/) | 79 points | by [timshell](https://news.ycombinator.com/user?id=timshell) | [63 comments](https://news.ycombinator.com/item?id=48324910)

Main idea: CAPTCHAs aren’t “broken”—AI can solve the images, but it doesn’t behave like humans while doing so. That behavioral gap is detectable.

What’s new:
- The authors introduce CogCAPTCHA30, a 30-task battery mixing a classic image CAPTCHA with 29 cognitive psychology tasks (decision-making, memory, perception, reasoning) to measure not just answers (output) but how they’re produced (process).
- They find output parity doesn’t imply process parity: humans and leading models hit similar accuracy, but differ in click sequences, direction changes, and overselection patterns. Across tasks, similarity in answers (Cohen’s d) is uncorrelated with similarity in process features (AUC).
- They propose a “Process Turing Test”: can a model’s behavior over time look human, not just its final answers?
- Surprisingly, frontier models (GPT, Claude, Gemini) look less human in their process traces than smaller models (Qwen) and a cognition-oriented model (Centaur). The authors attribute Centaur’s humanness to fine-tuning on 10M+ human choices across 160 cognitive experiments.

Adversarial angle:
- With full access to the detector’s features and objective, process-level fine-tuning can erase the gap.
- But the advantage fades when some features are hidden and disappears when models must generalize to new tasks, suggesting process-based detection can be robust if signals are diverse and undisclosed.

Why it matters:
- Site operators can revive CAPTCHAs by instrumenting behavior (click paths, timing, overselection) and rotating, hidden features across varied tasks.
- Expect an arms race: once process signals become targets, models can learn them—especially if signals leak. The durable path seems to be multi-task, cross-domain, and secret feature sets rather than static image puzzles.
