HN: Running a fast, local Gemma 4 coding agent with images on an M1 Max

TL;DR
- A developer got Gemma 4 26B running locally on a Mac (M1 Max, 64 GB) with llama.cpp + Metal, OpenAI-compatible API, and image support—fast enough for real-time coding-agent use.
- Multi-Token Prediction (MTP) via a Q8 draft model delivered ~24% faster generation vs. baseline.
- llama.cpp outperformed MLX in this setup; adding the multimodal projector enabled screenshots without slowing text generation.

What’s new
- Speculative decoding boost: With Gemma 4 26B-A4B (UD Q4_K_XL, ~16 GB) plus an MTP draft head (Q8_0), generation jumped from 58.2 tok/s to 72.2 tok/s on an M1 Max—about 1.24x faster. Prompt throughput stayed ~298 tok/s.
- Best MTP setting (hardware-dependent): --spec-draft-n-max 3 was fastest here (2 was close); higher values slowed down.
- llama.cpp vs. MLX: llama.cpp + Metal + MTP hit 72.2 tok/s; plain llama.cpp 58.2; MLX variants topped out around 45.8 tok/s. Despite MLX being Mac-focused, llama.cpp won in these tests.
- Images work: Adding the Gemma 4 multimodal projector (mmproj-BF16.gguf) let the agent accept screenshots via an OpenAI-compatible API (used with Pi). No observed slowdown in text generation with the projector loaded.

Why it matters
- Reliable offline dev: A fast, local coding agent that works without cloud connectivity, with tool compatibility via an OpenAI-style API.
- Practical speed: ~70 tok/s is “pleasantly usable” for agent workflows that churn through tool calls.
- Small-ish footprint: ~17 GB total for model + MTP head + projector.

Replicate in brief
- Build llama.cpp with Metal and Accelerate on macOS.
- Download:
  - Main: unsloth/gemma-4-26B-A4B-it-GGUF: gemma-4-26B-A4B-it-UD-Q4_K_XL.gguf
  - MTP draft: MTP/gemma-4-26B-A4B-it-Q8_0-MTP.gguf
  - Projector: mmproj-BF16.gguf
- Run llama.cpp with Metal and MTP:
  - Key flags: -ngl 999, -fa on, -c 4096, --model-draft <Q8 MTP>, --spec-type draft-mtp, --spec-draft-n-max 2–3
  - Add --mmproj mmproj-BF16.gguf for images
- Expose an OpenAI-compatible endpoint (e.g., llama.cpp server) and configure your client (e.g., Pi) to allow ["text","image"] inputs.

Benchmarks (128-token generations, M1 Max, macOS 15.7.7)
- llama.cpp + Metal (Q4): 58.2 tok/s
- llama.cpp + Metal + MTP (Q4 main + Q8 draft): 72.2 tok/s
- MLX (4-bit): 38–46 tok/s range
- Best --spec-draft-n-max: 3 on this machine

Caveats
- MTP gains vary by hardware; sweep --spec-draft-n-max (1–6) for your setup.
- Only Gemma 4 12B is natively multimodal; 26B needs the projector for images.
- Some MLX checkpoints may have loader/key mismatches; llama.cpp was smoother here.

**The Context:** A developer shared a guide on running a fast, multimodal coding agent locally using Gemma 4 (26B) on an M1 Max Mac. By leveraging `llama.cpp`, Multi-Token Prediction (MTP) draft models, and an OpenAI-compatible API, they achieved a highly usable ~72 tokens per second. 

Here is what the HN community had to say about the setup, benchmarking validity, and the current state of local macOS AI. 

#### **1. The Benchmark Debate: 128 Tokens is Too Short**
Several commenters pushed back against the original author’s 128-token benchmark. 
* **False Speedups:** Commenters noted that MTP (speculative decoding) speedups depend heavily on the acceptance rate of predicted tokens. Very short prompts and outputs can yield "false positive" speedups that don't reflect real-world usage.
* **Realistic Testing:** Users recommended testing context prompts in the 1,000 to 3,000 token range, and extending benchmarks to test context depth up to 32k or 64k tokens. 
* **Built-in Tooling:** Rather than writing custom scripts, commenters highly recommended using `llama-bench` (included natively in `llama.cpp`) to accurately profile token generation without stopping and starting the server.

#### **2. Quality-of-Life Tips for `llama.cpp`**
Users pointed out that downloading models is much easier now than the original post suggested:
* **Ditch the HuggingFace CLI:** You no longer need to use `huggingface-cli` to download models separately. You can simply pass the `-hf` flag directly to the `llama-server` command (e.g., `-hf unsloth/gemma-4-26B-A4B-it-GGUF`), and `llama.cpp` will download and cache the model automatically. 
* **Handling Images:** This same CLI approach works for downloading both the MTP draft models and the multimodal projector (`--mmproj`) needed for image processing. 

#### **3. Apple Silicon Quirks: MLX vs. GGUF**
A prominent discussion emerged around why the author used `llama.cpp` (GGUF) instead of Apple’s native MLX framework.
* **The Hardware Divide:** While MLX is fantastic (with commenters praising tools like LM Studio and OMLX), multiple users confirmed that on **older Apple Silicon (M1/M2 Max)**, `llama.cpp` often outperforms MLX in raw token speed and quantization overhead. 
* **The Upgrade Factor:** MLX’s distinct advantages become much more apparent on newer hardware like the M3 and M4 chips, which handle MLX’s architecture far better. 

#### **4. The Catch with MTP (Speculative Decoding)**
While MTP yielded a ~24% speed boost here, the community highlighted instances where it falls short:
* **Dense vs. MoE Models:** MTP provides massive speedups for *dense* models (like Gemma 4 26B). However, commenters pointed out that it yields diminishing returns for Mixture of Experts (MoE) models. Because MoE models already have a low *active* parameter count, raw generation speed is already high, making the computational overhead of a draft model less "worth it."
* **Formatting Breakages:** The original author actually chimed in to admit they temporarily stopped using MTP because it occasionally drops stop-tokens or breaks markdown and opencode XML tags, causing the AI's internal "thinking" processes to bleed messily into the frontend UI. 

#### **5. Looking Ahead: DeepSeek-v4-Flash**
As is tradition on HN, the conversation naturally drifted to the bleeding edge. Several users reported massive success running **DeepSeek-v4-Flash** locally using tools like the `ds4` repository. Users with beefy M4 Max configurations (128GB) reported hitting 24 tokens/second on DS-v4, noting that it genuinely feels like having an offline, GPT-4 class orchestrator for complex tool-calling and coding tasks.

### Slightly reducing the sloppiness of AI generated front end

- The author set out to quickly generate decent-looking personal apps with AI and found most outputs felt like “style + slop”—no matter the aesthetic requested.
- One prompt consistently cut the “AI slop” vibe: “make it look like a Qt app.” To their eyes, the Qt pass removed most of the mush across multiple small apps.
- Test case: a 270‑to‑win–style electoral map using 2030 forecast shifts. Many style attempts felt off; the Qt-styled version looked clean and coherent. They’ve since translated other personal tools to “Qt style” with similar improvements.
- Hypothesis: Qt’s constrained, native‑widget look (system fonts, muted palette, standard controls, stricter spacing) is easy for models to reproduce and naturally limits the gradients, card soup, and odd spacing that read as slop.
- The post asks for feedback and other style systems or guidelines that AIs can reliably mimic without adding slop on top.

Tip to try: Prompt for “desktop Qt app style using standard widgets, system font, neutral palette, minimal shadows, and a tight spacing grid,” and avoid asking for trendy web effects (big radii, gradients, glass, heavy shadows).

**Hacker News Daily Digest: The End of "AI Slop" Frontend Design?**

**The Pitch:** 
If you’ve used AI to generate a quick web app or personal tool recently, you’re probably familiar with the "AI Slop" aesthetic. It’s an unavoidable mishmash of trendy web effects: unnecessary gradients, huge border radii, heavy drop shadows, and disorganized "card soup" layouts. 

One developer found a surprisingly simple prompt to kill the slop: **"Make it look like a Qt app."** By asking the LLM to use desktop Qt app styling—system fonts, neutral palettes, standard widgets, and tight spacing—the AI produces clean, readable, and highly coherent interfaces. The author is now translating all their personal tools to this constrained, native-UI look and took to Hacker News to ask for other dependable design prompts. 

**What the Community is Saying:**
The Hacker News discussion quickly evolved into an analysis of *why* AI design defaults look the way they do, and how developers can outsmart the models to build better UIs.

Here are the top takeaways from the thread:

*   **AI Slop is the new Bootstrap:** Several commenters noted that the generic AI aesthetic isn’t necessarily "bad" design, but rather the modern equivalent of early Twitter Bootstrap. Just as default Bootstrap used to scream "low-effort side project" a decade ago, default AI generation now triggers the same reaction. 
*   **Why the "Qt Trick" works:** One insightful user pointed out that models excel at generating Qt designs simply because of the sheer volume of training data. Qt has been around for decades, meaning the internet—and by extension, the AI's training set—is packed with UI screenshots, tutorials, code snippets, and discussions that form a highly coherent "latent space" for the model to draw from.
*   **Nostalgia as a UI Framework:** Want another strict design constraint? Ask the AI to build a **Windows 95/9x app**. Commenters suggested that because the Win9x era relied on strict visual hierarchies, standard GDI controls, and simple 3D borders, LLMs can emulate it perfectly without getting lost in modern CSS effects.
*   **Just use an existing design system:** Why ask the AI to invent a component library from scratch? Some developers argued the simplest solution is to instruct the model to strictly adhere to established design systems like Material UI (MUI), GTK, or even point it toward classic CSS Zen Garden examples. 
*   **The "Squint Test":** Designers in the thread pointed out that AI often fails the classic "squint test" (blurring your eyes to see if the visual hierarchy and spacing still make sense). While LLMs excel at syntax, they still struggle to spatially reason about how structural elements group together on a screen. 
*   **Function over Form:** Despite HN's traditional disdain for generic AI code and design, several users shared success stories of non-technical friends using tools like Claude to build, deploy, and maintain incredibly useful internal tools (like SSL certificate monitoring scripts). For many, the "slop" doesn't matter if the app works flawlessly. 

**💡 Daily Tip:** 
Next time you're prompting an AI to build a frontend, avoid asking for trendy web effects. Instead, try this tested prompt formula from the original author: *"Use desktop Qt app style using standard widgets, a system font, neutral palette, minimal shadows, and a tight spacing grid."*

### Don't let the LLM speak, just probe it

#### [Submission URL](https://blog.j11y.io/2026-06-10_hidden-state-probes/) | 42 points | by [gmays](https://news.ycombinator.com/user?id=gmays) | [3 comments](https://news.ycombinator.com/item?id=48498283)

Don’t let the LLM speak—read its mind first. This post argues that when you ask an LLM “does this text satisfy this criterion?”, the decision is already encoded in the hidden state at the final prompt token. Instead of generating an answer, you grab that hidden state (often best ~midway up the stack), pass it to a tiny MLP or even a linear probe, and output a calibrated probability.

- Why: Need fast, cheap, calibrated judgments on structural questions (e.g., is the speaker themselves struggling? Is this sarcasm?), which embeddings miss and LLM “judges” make slow, costly, and vibe-based.
- How: Use a fixed prompt template ending with a seed token (e.g., “Assessment:”). Run (criterion, content, label) triples through a small open LLM (few-billion params), extract the hidden state at the seed, and train a small head. Vary the criteria during training so the head learns “does content satisfy criterion?” generally, not any specific rule. Calibrate with isotonic regression so 0.7 ≈ 70%.
- Optional LoRA: Finetune the backbone not to classify but to write one-sentence verdicts given known labels. At inference, never generate—the training just sharpens the geometry so the probe can read the decision cleanly.
- Results: One frozen model + one tiny head become an “any English classifier” with tens-of-milliseconds latency and embedding-like cost, working over long contexts—no per-criterion finetune.
- Key insight: The last token is the “address” where the model writes its decision; mid-layer reads often beat the final layer. This echoes linear probes, the logit lens, NLI cross-encoders (but with bigger context/reasoning), and RLHF reward heads that already score the final-token state.

The main debate in the comments centers on whether this technique offers a substantial advantage over simply prompting or LoRA-fine-tuning a model to output a single "yes" or "no" token. Commenters (*sths* and *wren6991*) pointed out that a single-token generation still only requires a single forward pass and can easily utilize KV caching for the shared parts of the prompt. ### The Normalization of Deviance in AI

#### [Submission URL](https://embracethered.com/blog/posts/2025/the-normalization-of-deviance-in-ai/) | 20 points | by [gurjeet](https://news.ycombinator.com/user?id=gurjeet) | [3 comments](https://news.ycombinator.com/item?id=48499790)

Title: The normalization of deviance in AI: treating LLMs as “trusted” is how systems fail

TL;DR: The AI industry is sleepwalking into a Challenger-style cultural failure—quietly normalizing the over-trust of LLM outputs (especially in agents) because “it worked last time.” LLM outputs are inherently untrusted and require downstream controls; skipping them turns benign mistakes into exploitable failures.

Key points:
- Core idea: “Normalization of deviance” (Diane Vaughan) = when repeated success despite rule-bending makes the deviation feel safe. In AI, this looks like treating probabilistic, non-deterministic LLMs as reliable components.
- Trust boundary: LLMs are unreliable actors. Outputs must be handled as untrusted data with access checks, sanitization/encoding, and other controls applied downstream.
- Drift in practice: As agentic systems complete tasks without visible failure, teams reduce oversight, loosen guardrails, and accept insecure defaults—confusing absence of incidents with presence of safety.
- Failure modes:
  - Benign errors: hallucinations, context loss, brittleness cause costly mistakes (e.g., wiping a prod DB, formatting drives, spamming issues).
  - Adversarial exploitation: indirect prompt injection and model backdoors can abuse agents and tools. Research shows small amounts of poisoned data can implant backdoors; triggers could be time- or content-based and transferable across the centralized LLM ecosystem.
- Vendor warnings underscore the risk:
  - Microsoft’s Agentic OS docs flag prompt injection and unintended actions, noting agents can behave like insider threats.
  - OpenAI’s ChatGPT Atlas cautions against use in regulated/sensitive/production contexts due to unresolved security risks.
  - Recent work from Anthropic and UCL highlights goal-seeking agent behaviors that can lead to harmful actions.
- Cultural drivers: Competitive pressure, cost-cutting, and hype incentivize shortcuts that become the new baseline; over time teams forget why guardrails existed.

Takeaways:
- Treat LLM output as untrusted by default. Keep a hard trust boundary, enforce least privilege on tools, validate/sanitize all actions, and maintain human oversight in high-stakes flows.
- Don’t mistake “it hasn’t failed yet” for “it’s safe.” The same drift that tolerates benign errors enables serious exploitation at scale.
